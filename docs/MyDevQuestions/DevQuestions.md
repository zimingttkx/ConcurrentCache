# 开发问题汇总

## 日志相关

### 什么是日志的异步写入？

**同步写入**：程序执行一条日志输出后，必须等待这条日志真正写入磁盘（或至少写入操作系统缓冲区）才能继续执行下一条代码。就像你寄信后必须等邮递员把信送到邮局才能做其他事。

**异步写入**：程序执行日志输出后，把日志丢到一个队列里就立即返回继续执行其他代码，后台有一个专门的线程负责慢慢把队列里的日志写入磁盘。就像你把信扔进邮筒就走了，邮递员什么时候来取、什么时候送走，你都不管。

---

### 为什么需要异步写入？

日志写入涉及 I/O 操作（磁盘很慢），如果用同步方式：
- 高并发时，日志写入会阻塞主业务线程
- 大量日志会拖慢整体性能

异步写入可以让主线程"只管写、不管写到哪里"，提高吞吐量。

---

### 同步 vs 异步对比

| | 同步 | 异步 |
|---|---|---|
| 像 | 在银行柜台排队办业务 | 在银行柜台挂号，然后去喝咖啡等叫号 |
| 日志时 | `log("msg");` → 等待写入完成 → 继续 | `log("msg");` → 立即返回 → 继续 |
| 缺点 | 慢 | 可能丢日志（如果程序崩溃，队列里的还没写完） |

---

### 常见实现方式

1. **双缓冲**：准备两个缓冲区，A缓冲区接收日志时，B缓冲区负责写入；满了就交换
2. **队列**：生产者（主线程）往队列塞日志，消费者（后台线程）从队列取日志写入
3. **环形缓冲区**：固定大小的环形队列，生产和消费指针循环移动

---

## 日志轮转（Log Rotation）

日志系统运行久了，日志文件会越来越大，可能把磁盘撑满。**日志轮转**就是解决这个问题的机制——把日志按大小或时间切成多个文件，旧日志归档，新的继续写。

### 举个例子

假设你有一个 `app.log`：

```
app.log          ← 当前日志，正在写入 (5MB)
app.log.1        ← 昨天的日志 (20MB)
app.log.2        ← 前天的日志 (20MB)
app.log.3        ← 大前天的日志 (20MB)
```

当日志文件达到设定大小（比如20MB）时：
1. 把 `app.log.1` 重命名为 `app.log.2`
2. 把 `app.log` 重命名为 `app.log.1`
3. 创建一个新的空 `app.log` 继续写

这就是"轮转"——像转盘一样，旧的往外推，新的在中心写。

### 常见策略

| 策略 | 说明 |
|---|---|
| **按大小** | 达到指定大小就轮转（如 100MB） |
| **按时间** | 每天/每小时轮转一次 |
| **按日期** | 每天一个日志文件（如 `app.2026-04-24.log`） |
| **压缩旧日志** | 旧日志 gzip 压缩节省空间 |

### 为什么需要？

- 防止单个日志文件无限增长撑满磁盘
- 方便查找特定时间的日志
- 旧日志可以定期清理或归档

## C++ 条件变量 wait 与 lambda

### 标准写法（看得懂就行）

```cpp
cv_.wait(lock, [this] { return queue_.empty(); });
```

这是 `std::condition_variable` 的第二种重载，加了一个返回 bool 的条件。

---

### 拆解：wait 在干什么

`wait(lock, 条件)` 等价于：

```cpp
// 伪代码，实际上 while 循环 + wait 是原子的
while (!条件) {
    cv_.wait(lock);  // 解锁 → 阻塞等待通知 → 加锁 → 继续循环
}
```

也就是说：
1. **条件为 true** → 直接返回，后面的代码继续执行
2. **条件为 false** → 解锁锁，然后阻塞。等别人 `notify_one()` 叫醒它后，加锁，再检查条件。为 false 就继续阻塞

---

### lambda 是什么

就是**匿名函数**，当场定义、当场用，不用单独写个函数。

```cpp
// 普通函数
bool is_empty() {
    return queue_.empty();
}

// lambda 版本
[this] { return queue_.empty(); }
```

---

### 捕获列表 `[this]` 是什么意思

lambda 里想用外面的变量，就得捕获。

| 写法 | 含义 |
|---|---|
| `[]` | 不捕获任何变量，函数里用不了外面的东西 |
| `[this]` | 捕获 this 指针，能用成员变量 `queue_` |
| `[&]` | 按引用捕获所有外部变量 |
| `[=]` | 按值捕获所有外部变量 |

```cpp
int x = 10;

[=]() { return x; }  // 可以用 x，按值拷贝
[&]() { return x; }   // 可以用 x，按引用（x 变了这里也跟着变）
[x]() { return x; }  // 只捕获 x，按值
```

日志队列场景用 `[this]` 是因为 lambda 里要访问成员变量 `queue_`。

---

### 两种写法对比

```cpp
// 写法1：while 循环（老式，看做了解就行）
while (!queue_.empty()) {
    cv_.wait(lock);
}

// 写法2：wait + lambda（推荐，写起来短）
cv_.wait(lock, [this] { return queue_.empty(); });
```

两者效果完全一样，lambda 版本更简洁。

---

### 实际例子：生产者-消费者

```cpp
// 生产者：往队列塞数据，然后通知
void push(const string& msg) {
    {
        lock_guard<mutex> lock(mtx_);
        queue_.push(msg);
    }
    cv_.notify_one();  // 通知一个等待的线程
}

// 消费者：等队列有数据才取
string pop() {
    cv_.wait(lock, [this] { return !queue_.empty(); });  // 等待直到非空
    string msg = queue_.front();
    queue_.pop();
    return msg;
}
```

这就是日志异步写入的核心：**主线程往队列塞日志（push），后台线程从队列取日志写入文件（pop）**。

---

### 常见的坑

1. **忘了 notify**：push 之后一定要 `notify_one()` 或 `notify_all()`，否则 pop 会一直卡住
2. **在锁外面 notify**：上面的例子用大括号包住临界区，这样 notify 是在锁外面发的，避免死锁
3. **虚假唤醒**：wait 被系统莫名其妙叫醒，但条件其实没满足。lambda 版本自动重新检查条件，所以没问题；如果是 while 循环手动 wait，就要用 while 而不是 if
