# 第一章：概述与设计理念

## 1.1 项目背景

### 1.1.1 为什么要开发这个项目？

在互联网后端开发中，**缓存**是提升系统性能的关键技术。Redis 作为最流行的内存数据库，被广泛应用于各大互联网公司。

```
没有缓存：
用户请求 ──► 数据库查询 ──► 返回结果
                    │
                    ▼ 慢！数据库压力大！

有缓存：
用户请求 ──► 缓存查找 ──► 命中！返回
                              │
                              ▼ 未命中
                        数据库查询 ──► 写入缓存 ──► 返回
```

但是有时候：
1. **只是想学习 Redis 的工作原理**
2. **项目太小，不需要引入完整的 Redis**
3. **想要一个可以嵌入的轻量级缓存**

于是你决定：**自己写一个简单的缓存服务器**。

### V1 约束与挑战

| 约束 | 说明 | 应对 |
|------|------|------|
| 单线程 Reactor | 只能使用一个 CPU 核心 | 先实现功能，后续版本升级多线程 |
| 简单存储 | 只用 unordered_map | 先能用就行，后续版本加 TTL/LRU |
| 有限命令 | 只实现 4 个命令 | 先验证架构，后续版本扩展 |
| 无持久化 | 数据在内存，重启丢失 | 后续版本支持 RDB/AOF |

**设计原则**：先解决"能不能跑"的问题，再解决"跑得好不好"的问题。

## 1.5 核心设计理念

### 1.5.1 为什么选择事件驱动而不是多线程？

```
多线程方案的问题：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 线程1   │ │ 线程2   │ │ 线程3   │   ← 每个线程都要占用栈内存
│ 等待I/O │ │ 等待I/O │ │ 等待I/O │   ← 大部分时间在等，浪费CPU
└─────────┘ └─────────┘ └─────────┘

事件驱动方案：
┌─────────────────────────────────────┐
│           EventLoop (单线程)        │
│                                     │
│   epoll_wait() 阻塞等待 ──► 有事才处理
│                                     │
│   处理 A ──► 处理 B ──► 处理 C      │  ← CPU 始终在工作
└─────────────────────────────────────┘
```

**结论**：

| 场景 | 推荐方案 |
|------|----------|
| IO 密集型（网络服务器） | 事件驱动 ✓ |
| CPU 密集型（计算任务） | 多线程 |
| 混合型 | 事件驱动 + 线程池 |

### 1.5.2 为什么要分层？

```
不分层的问题：
- 全部代码混在一起
- 改网络逻辑要动业务代码
- 难以测试、难以复用
- 扩展新命令很麻烦

分层后的优势：
- 每层只管自己的事
- 可以独立测试
- 可以单独替换
- 易于理解和维护
```

### 1.5.3 为什么要用 RAII 和智能指针？

```
手动管理内存的问题：
- 忘记 delete → 内存泄漏
- 重复 delete → 崩溃

RAII + 智能指针：
- 对象生命周期绑定到作用域
- 作用域结束时自动释放
- 不用担心忘记或重复释放
```

## 1.6 请求处理流程（SET name zhangsan）

```
1. 客户端发送 "SET name zhangsan\r\n"
   │
   ▼
2. TCP 数据到达服务器的 socket
   │
   ▼
3. epoll 检测到 socket 可读
   │
   ▼
4. EventLoop 回调 Connection::handle_read()
   │
   ▼
5. Connection 从 socket 读取数据到 Buffer
   │
   ▼
6. RespParser 从 Buffer 解析出命令
   │
   ▼
7. CommandFactory 根据命令名创建具体命令对象
   │
   ▼
8. 命令对象执行：从 GlobalStorage 读写数据
   │
   ▼
9. RespEncoder 将执行结果编码成响应
   │
   ▼
10. Connection 将响应写入 Buffer
    │
    ▼
11. epoll 检测到 socket 可写
    │
    ▼
12. EventLoop 回调 Connection::handle_write()
    │
    ▼
13. Connection 将响应发送给客户端
```
