# 第三章：协议层（RESP 协议）

## 章节预告

上一章我们看到，应用层需要处理命令参数 `["SET", "name", "zhangsan"]`。

但问题是：**这些数据从哪里来？**

网络层接收的是一串字节流：
```
*3\r\n$3\r\nSET\r\n$4\r\nname\r\n$8\r\nzhangsan\r\n
```

这串字节是如何变成我们能理解的命令数组的？这就是**协议层**要解决的问题。

---

## 3.1 问题：TCP 是流协议

### 3.1.1 TCP 的特点

TCP（Transmission Control Protocol）是一种**面向流的协议**：

```
UDP 的工作方式（面向消息）：
┌──────────┐      ┌──────────┐
│  消息 A  │ ───► │  消息 A  │
└──────────┘      └──────────┘
     │                  │
┌──────────┐      ┌──────────┐
│  消息 B  │ ───► │  消息 B  │
└──────────┘      └──────────┘

特点：消息边界完整保留，收到一条就是一条


TCP 的工作方式（面向流）：
┌──────────┐      ┌──────────┐
│ 数据流   │ ───► │ 数据流   │
└──────────┘      └──────────┘

特点：像水管里的水，没有边界，需要自己判断消息在哪里结束
```

### 3.1.2 粘包问题

**粘包** = 多个消息粘在一起，分不清边界

```
发送端连续发送：
┌─────────┐┌─────────┐┌─────────┐
│ 命令 A  ││ 命令 B  ││ 命令 C  │
└─────────┘└─────────┘└─────────┘

接收端可能收到：
┌─────────────────────┐
│ A │ B │C│           │ ← A、B、C 粘在一起
└─────────────────────┘

或者：
┌─────────────────────┐
│ A │B 命令合并 │     │ ← 部分合并
└─────────────────────┘

或者：
┌─────────┐┌─────────┐
│ A │BC  │             │ ← 分拆
└─────────┘└─────────┘
```

### 3.1.3 半包问题

**半包** = 一个消息被拆成多次接收

```
发送端发送一个大消息（比 TCP 缓冲区还大）：
┌─────────────────────────────┐
│         大命令             │
└─────────────────────────────┘

接收端分多次收到：
┌─────────┐┌─────────┐┌─────────┐
│  第1片  ││  第2片  ││  第3片  │
└─────────┘└─────────┘└─────────┘
```

### 3.1.4 解决方案：应用层协议

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           解决方案                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  粘包和半包问题是 TCP 的固有特性，无法避免。                                │
│  但我们可以在应用层定义协议，自己解决这些问题。                              │
│                                                                             │
│  常见的应用层协议设计方式：                                                  │
│                                                                             │
│  1. 固定长度：每个消息都是 100 字节，不足补空格                            │
│     └─ 简单但浪费带宽                                                     │
│                                                                             │
│  2. 分隔符：以 \r\n 结尾（如 Redis 的 RESP 协议）                        │
│     └─ 灵活高效，需要扫描查找分隔符                                        │
│                                                                             │
│  3. 长度前缀：先发长度，再发内容                                          │

│                                                                             │
│  4. 混合模式：用长度描述二进制内容（如 HTTP）                             │
│     └─ 功能强大，解析复杂                                                  │
│                                                                             │

│                                                                             │

```

## 3.2 RESP 协议详解

### 3.2.1 RESP 是什么？

RESP = **RE**dis **S**erialization **P**rotocol

RESP 是 Redis 客户端和服务端通信的协议，是 Redis 的灵魂。

**为什么选择 RESP？**
```
1. 文本协议，人类可读，便于调试
2. 简单高效，解析速度快
3. 支持 Redis 客户端直接连接
4. 协议设计优秀，值得学习
```

### 3.2.2 RESP 数据类型

RESP 定义了 5 种数据类型：

```
┌─────────────────────────────────────────────────────────────────────────────┐

│     首字符: +                                                            │

│     用于: 表示成功响应                                                     │
│                                                                             │
│  2. 错误 (Error)                                                         │

│     示例: -ERR unknown command\r\n                                        │
│     用于: 表示错误信息                                                     │

│     用于: 表示整数值                                                       │

│  4. 批量字符串 (Bulk String)                                             │
```

## 3.3 协议解析器实现

### 3.3.1 RespParser 类

**文件：** `src/protocol/resp.h`

```cpp
class RespParser {
public:
    // 核心解析函数
    std::vector<RespValue> parse(Buffer* buffer);

    // 重置解析器状态
    void reset();

    // 获取错误信息
    const std::string& error() const { return error_msg_; }

    // 检查是否有完整命令
    static bool has_complete_command(const Buffer* buffer);

private:
    // 解析单个 RESP 值
    bool parse_one(Buffer* buffer, RespValue& out);

    // 解析各类数据
    bool parse_simple_string(Buffer* buffer, std::string& out);
    bool parse_error(Buffer* buffer, std::string& out);
    bool parse_integer(Buffer* buffer, int64_t& out);
    bool parse_bulk_string(Buffer* buffer, std::string& out);
    bool parse_array(Buffer* buffer, std::vector<RespValue>& out);

    // 工具函数
    static const char* find_crlf(const char* data, size_t len);
    static void skip(Buffer* buffer, size_t len);

    std::string error_msg_;
};
```

### 3.3.2 解析核心：parse() 函数

```cpp
std::vector<RespValue> RespParser::parse(Buffer* buffer) {
    std::vector<RespValue> commands;

    // 循环解析，直到没有完整命令
    while (has_complete_command(buffer)) {
        RespValue cmd;

        if (parse_one(buffer, cmd)) {
            commands.push_back(cmd);  // 解析成功，添加到结果
        } else {
            break;  // 解析失败，数据不完整，保留在 buffer 中
        }
    }

    return commands;
}
```

### 3.3.3 解析单个值：parse_one()

```cpp
bool RespParser::parse_one(Buffer* buffer, RespValue& out) {
    const char* data = buffer->peek();
    size_t len = buffer->readable_bytes();

    if (len < 1) {
        error_msg_ = "Buffer is empty";
        return false;
    }

    // 根据首字符判断数据类型
    char type = data[0];

    switch (type) {
        case '+': {
            std::string content;
            if (!parse_simple_string(buffer, content)) return false;
            out.type = RespType::SIMPLE_STRING;
            out.data = content;
            return true;
        }
        case '-': {
            std::string content;
            if (!parse_error(buffer, content)) return false;
            out.type = RespType::ERROR;
            out.data = content;
            return true;
        }
        case ':': {
            int64_t value = 0;
            if (!parse_integer(buffer, value)) return false;
            out.type = RespType::INTEGER;
            out.data = value;
            return true;
        }
        case '$': {
            std::string content;
            if (!parse_bulk_string(buffer, content)) return false;
            out.type = RespType::BULK_STRING;
            out.data = content;
            return true;
        }
        case '*': {
            std::vector<RespValue> arr;
            if (!parse_array(buffer, arr)) return false;
            out.type = RespType::ARRAY;
            out.data = arr;
            return true;
        }
        default:
            error_msg_ = "Unknown RESP type: " + std::string(1, type);
            return false;
    }
}
```

### 3.3.4 解析批量字符串

批量字符串格式：`<length>\r\n<content>\r\n`

```cpp
bool RespParser::parse_bulk_string(Buffer* buffer, std::string& out) {
    const char* data = buffer->peek();
    size_t len = buffer->readable_bytes();

    // 步骤1：找到长度行的 \r\n
    const char* crlf = find_crlf(data + 1, len - 1);
    if (!crlf) {
        error_msg_ = "Bulk string missing length CRLF";
        return false;
    }

    // 步骤2：解析长度
    std::string len_str(data + 1, crlf - (data + 1));
    int64_t bulk_len = std::stoll(len_str);

    // 步骤3：计算头部长度（"$<len>\r\n"）
    size_t header_len = (crlf - data) + 2;

    // 步骤4：处理 null（长度 -1）
    if (bulk_len < 0) {
        skip(buffer, header_len);  // 只跳过 $-1\r\n
        out = "";
        return true;
    }

    // 步骤5：检查内容是否完整
    // 完整长度 = 头部长度 + 内容长度 + \r\n
    size_t total_len = header_len + bulk_len + 2;

    if (len < total_len) {
        error_msg_ = "Bulk string incomplete";
        return false;
    }

    // 步骤6：提取内容
    out = std::string(data + header_len, bulk_len);

    // 步骤7：跳过已解析的数据
    skip(buffer, total_len);
    return true;
}
```

### 3.3.5 解析数组

数组格式：`*<count>\r\n<element1>...<elementN>`

```cpp
bool RespParser::parse_array(Buffer* buffer, std::vector<RespValue>& out) {
    const char* data = buffer->peek();
    size_t len = buffer->readable_bytes();

    // 步骤1：解析数组计数行
    const char* crlf = find_crlf(data + 1, len - 1);
    if (!crlf) {
        error_msg_ = "Array missing count CRLF";
        return false;
    }

    std::string count_str(data + 1, crlf - (data + 1));
    int64_t count = std::stoll(count_str);

    // 跳过 *<count>\r\n
    skip(buffer, (crlf - data) + 2);

    // 步骤2：处理 null 数组（count < 0）
    if (count < 0) {
        return true;
    }

    // 步骤3：循环解析每个元素
    out.reserve(static_cast<size_t>(count));

    for (int64_t i = 0; i < count; ++i) {
        if (buffer->readable_bytes() < 1) {
            error_msg_ = "Array element missing";
            return false;
        }

        // 递归解析每个元素
        RespValue elem;
        if (!parse_one(buffer, elem)) {
            return false;
        }

        out.push_back(elem);
    }

    return true;
}
```

## 3.5 Buffer 与解析的协作

### 3.5.1 Buffer 的双指针设计

Buffer 是 TCP 读写缓冲区，解决了粘包问题：

```cpp
class Buffer {
private:
    std::vector<char> buffer_;    // 存储数据的 vector
    size_t reader_idx_;          // 读指针：已读到哪里
    size_t writer_idx_;          // 写指针：已写到哪里

public:
    // 可读字节数
    size_t readable_bytes() const {
        return writer_idx_ - reader_idx_;
    }

    // 可写字节数
    size_t writable_bytes() const {
        return buffer_.size() - writer_idx_;
    }

    // 追加数据
    void append(const char* data, size_t len) {
        ensure_writable(len);
        std::copy(data, data + len, buffer_.begin() + writer_idx_);
        writer_idx_ += len;
    }

    // 读取数据（不删除）
    const char* peek() const {
        return buffer_.data() + reader_idx_;
    }

    // 标记已读
    void retrieve(size_t len) {
        if (len > readable_bytes()) {
            reader_idx_ = writer_idx_;  // 超过可读范围，归零
        } else {
            reader_idx_ += len;
        }

        // 如果已读超过一半，自动压缩
        if (reader_idx_ > buffer_.size() / 2) {
            compact();
        }
    }

    // 压缩空间（把未读数据移到开头）
    void compact() {
        if (reader_idx_ == 0) return;
        std::copy(
            buffer_.begin() + reader_idx_,
            buffer_.begin() + writer_idx_,
            buffer_.begin()
        );
        writer_idx_ -= reader_idx_;
        reader_idx_ = 0;
    }
};
```

### 3.5.2 Buffer 工作原理

```
初始状态：
┌─────────────────────────────────────────────────────────────┐
│ [ 已读区域 ]  │ [ 已读区域 ]  │ [ 可写区域 ]                │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity

写入 "SET" 后：
┌─────────────────────────────────────────────────────────────┐
│              │  S  E  T          │ [ 可写区域 ]            │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity

读取 peek() 后（指针不变）：
┌─────────────────────────────────────────────────────────────┐
│              │  S  E  T          │ [ 可写区域 ]            │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity
返回 "SET" 的指针

读取 retrieve(3) 后（移动读指针）：
┌─────────────────────────────────────────────────────────────┐
│ [ 已读区域 ]  │              │ [ 可写区域 ]                │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity

写入 "GET" 后：
┌─────────────────────────────────────────────────────────────┐
│ [ 已读 ]  │  G  E  T       │ [ 可写区域 ]                │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity
```

### 3.5.3 粘包解决方案

```
场景1：多个命令一起到达

发送：
"GET name\r\nSET age 25\r\n"

接收端 Buffer：
┌─────────────────────────────────────────────────────────────┐
│ G  E  T     n  a  m  e  \r  \n  S  E  T     a  g  e     │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑           ↑
             reader_idx    writer_idx    capacity

解析第一个命令 (has_complete_command 检测到 \r\n)：
  - parse_one() 解析 "GET name\r\n"
  - retrieve() 移动读指针

解析第二个命令：
  - has_complete_command 再次检测
  - parse_one() 解析 "SET age 25\r\n"


场景2：命令被分包

第1次接收："GET n"
第2次接收："ame\r\n"

第1次接收后 Buffer：
┌─────────────────────────────────────────────────────────────┐
│ G  E  T     n                                      │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑
             reader_idx    writer_idx

has_complete_command 检测：
  - 数据不够（没有 \r\n）
  - 返回 false，不解析

第2次接收后 Buffer：
┌─────────────────────────────────────────────────────────────┐
│ G  E  T     n  a  m  e  \r  \n                       │
└─────────────────────────────────────────────────────────────┘
                ↑              ↑
             reader_idx    writer_idx

has_complete_command 检测：
  - 找到 \r\n
  - 返回 true，开始解析
```

```
步骤1：has_complete_command 检测

```

## 3.7 RespValue 结构

### 3.7.1 设计

```cpp
// RESP 支持的数据类型
enum class RespType : char {
    SIMPLE_STRING = '+',
    ERROR = '-',
    INTEGER = ':',
    BULK_STRING = '$',
    ARRAY = '*',
    UNKNOWN = 0
};

// 使用 std::variant 存储不同类型的值
struct RespValue {
    // variant 可以存储多种类型之一
    std::variant<
        std::monostate,              // 空值（null）
        std::string,                  // 字符串
        int64_t,                     // 整数
        std::vector<RespValue>       // 数组
    > data;

    RespType type = RespType::UNKNOWN;

    // 便捷访问方法
    std::string as_string() const;
    int64_t as_integer() const;
    const std::vector<RespValue>& as_array() const;
    bool is_null() const;
};
```

### 3.7.2 使用示例

```cpp
RespValue val = parse_result;

// 判断类型
if (val.type == RespType::ARRAY) {
    // 获取数组
    const auto& arr = val.as_array();
    for (const auto& elem : arr) {
        // 处理每个元素
    }
}

// 或者使用 variant 的类型检查
if (std::holds_alternative<std::string>(val.data)) {
    std::string str = std::get<std::string>(val.data);
}
```

## 3.8 为什么这样设计？

### 3.8.1 设计原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           协议设计原则                                      │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 人类可读                                                               │
│     ─────────                                                              │
│     文本协议，便于调试和理解                                               │
│     $3\r\nGET\r\n 而不是二进制 0x01 0x02 ...                            │
│                                                                             │
│  2. 高效解析                                                               │
│     ─────────                                                              │
│     基于 \r\n 分隔，解析速度快                                            │
│     只需顺序扫描，不需要复杂的状态机                                       │
│                                                                             │
│  3. 二进制安全                                                             │
│     ───────────                                                            │
│     Bulk String 有长度前缀，可以包含任意二进制数据                        │
│                                                                             │
│  4. 类型丰富                                                               │
│     ───────                                                                │
│     支持 5 种类型，覆盖常见数据结构                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.8.2 权衡取舍

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           权衡取舍                                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  选择 RESP 而不是 HTTP/JSON 的原因：                                       │
│                                                                             │
│  ✓ RESP 更简单，专为 Redis 设计                                           │
│  ✓ 文本协议，调试方便                                                     │
│  ✓ 解析速度快                                                             │
│  ✓ Redis 客户端可直接测试                                                 │
│                                                                             │
│  ✗ 不是通用协议，换个客户端要重新实现                                      │
│  ✗ 二进制支持不如 MessagePack 等协议                                      │
│                                                                             │
│  但对于"学习目的"和"简单缓存"，RESP 是完美选择                          │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 3.9 本章小结

### 3.9.1 关键概念

| 概念 | 说明 |
|------|------|
| TCP 流协议 | 数据没有边界，需要应用层解决粘包 |
| 粘包 | 多个消息粘在一起 |
| 半包 | 一个消息被拆成多次接收 |
| RESP 协议 | Redis 的文本协议，5 种数据类型 |
| Buffer 双指针 | 用读写指针解决粘包问题 |

### 3.9.2 RESP 数据类型

| 类型 | 首字符 | 格式 |
|------|--------|------|
| 简单字符串 | + | +<content>\r\n |
| 错误 | - | -<content>\r\n |
| 整数 | : | :<number>\r\n |
| 批量字符串 | $ | $<len>\r\n<content>\r\n |
| 数组 | * | *<count>\r\n<elements> |

### 3.9.3 解析流程

```
1. Buffer 接收 TCP 数据
2. has_complete_command 检查是否有完整命令
3. parse_one 根据首字符判断类型
4. 递归解析子元素（数组）
5. retrieve 移动读指针
6. 返回 RespValue 结构
```
