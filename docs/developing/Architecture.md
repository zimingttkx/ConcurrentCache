## 优雅退出流程

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           优雅退出流程                                      │
│                                                                             │
│  1. 收到 SIGINT/SIGTERM                                                    │
│     ↓                                                                       │
│  2. signal_handler() 调用 g_main_reactor->event_loop()->quit()             │
│     ↓                                                                       │
│  3. quit() 设置 quit_=true 并调用 wakeup()                                 │
│     ↓                                                                       │
│  4. MainReactor 的 EventLoop 检测到 quit_，退出 epoll_wait                 │
│     ↓                                                                       │
│  5. main() 继续执行，依次调用：                                             │
│     - SubReactorPool::stop() - 停止所有 SubReactor                         │
│     - MainReactor::stop()   - 停止 MainReactor                             │
│     - ThreadPool::stop()    - 停止线程池                                    │
│     ↓                                                                       │
│  6. 进程安全退出                                                            │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 核心设计理念

### 1. 事件驱动 + 多线程（MainSubReactor）

```
单 Reactor 方案（V1）：
┌─────────────────────────────────────┐
│         EventLoop (单线程)          │
│                                     │
│   epoll_wait() 阻塞等待 ──► 有事才处理
│                                     │
│   accept() ──► 处理 A ──► 处理 B   │  ← 只能用到1核
└─────────────────────────────────────┘

MainSubReactor 方案（V2）：
┌─────────────────────────────────────┐
│         MainReactor (线程0)        │
│              accept()               │
└─────────────────┬───────────────────┘
                  │
                  │ 分发新连接
                  ▼
┌──────────┐ ┌──────────┐ ┌──────────┐
│ SubReactor│ │ SubReactor│ │ SubReactor│
│   线程1   │ │   线程2   │ │   线程3   │
│ ┌──────┐ │ │ ┌──────┐ │ │ ┌──────┐ │
│ │EventLoop│ │ │EventLoop│ │ │EventLoop│ │
│ │ 处理   │ │ │ 处理   │ │ │ 处理   │ │
│ │ 连接   │ │ │ 连接   │ │ │ 连接   │ │
│ └──────┘ │ │ └──────┘ │ │ └──────┘ │
└──────────┘ └──────────┘ └──────────┘
      │            │            │
      ▼            ▼            ▼
   CPU 核1      CPU 核2      CPU 核3  ← 全部利用！
```

### 2. 三级内存池

```
为什么需要内存池？

malloc 的问题：
- 系统调用开销（用户态↔内核态）
- 锁竞争（所有线程共享堆）
- 内存碎片（小对象频繁分配）

内存池的优势：
┌─────────────────────────────────────────────────────────────┐
│                    ThreadCache（第一层）                      │
│  thread_local，每个线程独立                                │
│  分配/释放完全无锁，极快                                   │
│  批量获取/归还，减少锁操作                                 │
└───────────────────────────────┬─────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────┐
│                    CentralCache（第二层）                    │
│  所有线程共享                                               │
│  每个 SizeClass 独立锁（细粒度锁）                          │
│  管理 Span，切分成固定大小的小块                             │
└───────────────────────────────┬─────────────────────────────┘
                                │
                                ▼
┌─────────────────────────────────────────────────────────────┐
│                    PageCache（第三层）                       │
│  直接和系统交互（mmap/munmap）                              │
│  管理 4KB 页                                               │
│  Span 合并减少碎片                                          │
└─────────────────────────────────────────────────────────────┘
```

### 3. 线程池

```
为什么需要线程池？

每任务一线程的问题：
- 1000个并发请求 = 1000个线程
- 每个线程占用1MB+栈内存
- 线程创建/销毁开销大

线程池解决方案：
┌─────────────────────────────────────────────────────────────┐
│                    ThreadPool                               │
│  主线程 ────── 提交任务 ──────► 阻塞队列                   │
│      │                              │                       │
│      │◄──── 返回 future ───────────┘                       │
│      │                              │                       │
│      │                    ┌─────────┼─────────┐            │
│      │                    ▼         ▼         ▼            │
│      │              工作线程1  工作线程2  工作线程3         │
└─────────────────────────────────────────────────────────────┘
```

### 4. 分层架构

```
应用层：命令处理（用户能看到的）
    │
协议层：数据解析（如何理解数据）
    │
存储层：数据存储（如何保存数据）
    │
网络层：通信传输（如何收发数据）
    │
基础层：日志、配置、信号、锁、内存池、线程池（工程化基础设施）
```

## 请求处理流程（SET name zhangsan）

```
1. 客户端发送 "SET name zhangsan\r\n"
   │
   ▼
2. TCP 数据到达服务器的 socket
   │
   ▼
3. MainReactor 的 epoll_wait 检测到 listen socket 可读
   │
   ▼
4. MainReactor 调用 accept() 接受连接
   │
   ▼
5. SubReactorPool 轮询选择一个 SubReactor
   │
   ▼
6. SubReactor 的 EventLoop 注册新连接的 channel
   │
   ▼
7. 客户端发送数据，SubReactor 的 epoll_wait 检测到可读
   │
   ▼
8. SubReactor 回调 Connection::handle_read()
   │
   ▼
9. Connection 从 socket 读取数据到 Buffer
   │
   ▼
10. RespParser 从 Buffer 解析出命令
    │
    ▼
11. CommandFactory 根据命令名创建具体命令对象
    │
    ▼
12. 命令对象执行：从 GlobalStorage 读写数据
    │
    ▼
13. RespEncoder 将执行结果编码成响应
    │
    ▼
14. Connection 将响应写入 Buffer
    │
    ▼
15. SubReactor 的 epoll_wait 检测到 socket 可写
    │
    ▼
16. SubReactor 回调 Connection::handle_write()
    │
    ▼
17. Connection 将响应发送给客户端
```
