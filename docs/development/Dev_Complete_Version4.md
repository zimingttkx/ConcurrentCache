# 第五章：分布式版本（Version4）

## 版本概述

**版本目标**：实现分布式集群功能，支持水平扩展和高可用。

**问题分析**：增强版本是单机版的终极形态，但存在明显的分布式问题：

```
增强版本的问题分析：
┌─────────────────────────────────────────────────────────────┐
│  问题1：单机存储容量受内存限制                               │
│  ─────────────────────────────────────────────────         │
│  所有数据都存在一台机器的内存中                             │
│  → 4GB内存 → 最多存4GB数据                                │
│  → 海量数据无法存储                                       │
│  → 无法通过增加机器来扩展容量                              │
│                                                              │
│  问题2：单点故障                                           │
│  ─────────────────────────────────────────────────         │
│  只有一台机器提供服务                                       │
│  → 机器崩溃 → 服务中断                                    │
│  → 断电 → 服务中断                                        │
│  → 数据丢失                                               │
│                                                              │
│  问题3：无法水平扩展                                      │
│  ─────────────────────────────────────────────────         │
│  只能垂直升级硬件（换更大内存、更多CPU）                    │
│  → 成本呈指数增长                                         │
│  → 有物理上限                                             │
└─────────────────────────────────────────────────────────────┘
```

**为什么需要分布式版本**：

| 需求 | 说明 |
|-----|------|
| **水平扩展** | 通过增加节点来扩展容量和性能 |
| **高可用** | 主从复制保证数据不丢失，自动故障转移保证服务连续性 |
| **数据分片** | 16384个槽分散到多个节点，充分利用集群资源 |
| **负载均衡** | 请求自动路由到合适的节点 |

**预计开发周期**：12-16周

### 5.2.2 槽映射管理

#### 为什么需要槽映射？

| 需求 | 说明 |
|-----|------|
| **数据分片** | 16384个槽分散在不同节点 |
| **键定位** | 通过CRC16算法快速计算键属于哪个槽 |
| **负载均衡** | 槽均匀分布保证各节点负载大致相同 |
| **扩缩容基础** | 槽迁移是扩缩容的基础 |

#### CRC16算法原理

```
键 → CRC16(key) → 取模16384 → 槽号

CRC16（Redis标准多项式）：
- 输入：任意长度的key字符串
- 输出：0-65535的值
- 取模16384得到槽号

示例：
"user:1000" → CRC16 → 12345 → 12345 % 16384 → 槽12345
```

#### 槽分配示例

```
3节点集群的槽分配：
Node1: 槽0-5460     (5461个槽)
Node2: 槽5461-10922 (5462个槽)
Node3: 槽10923-16383 (5461个槽)

键"user:1000"计算得槽12345 → Node2处理
```

#### 实现要点

```cpp
class SlotMap {
private:
    std::array<std::string, 16384> slot_to_node_;  // 槽到节点的映射
    std::map<std::string, std::set<uint16_t>> node_to_slots_;  // 节点到槽的反向映射

public:
    // 键到槽的映射
    static uint16_t key_to_slot(const std::string& key) {
        // CRC16算法
        uint16_t crc = crc16(key.data(), key.size());
        return crc % 16384;
    }

    void set_slot_owner(uint16_t slot, const std::string& node_id) {
        slot_to_node_[slot] = node_id;
        node_to_slots_[node_id].insert(slot);
    }

    std::string get_slot_owner(uint16_t slot) {
        return slot_to_node_[slot];
    }
};
```

#### 验收标准

- 键能正确映射到槽
- 槽分配信息正确管理

### 5.3.2 Gossip协议

#### 为什么选择Gossip？

Gossip协议是分布式系统中常用的状态同步协议，类比"八卦传播"：

| 特性 | 说明 |
|-----|------|
| **去中心化** | 无需中心节点，每个节点都可以传播信息 |
| **最终一致** | 信息逐渐传播到所有节点 |
| **容错性** | 节点宕机不影响其他节点通信 |
| **可扩展性** | 节点越多，传播越均匀 |

#### Gossip传播示意图

```
初始：Node1知道新节点信息
       Node1
         ↓ Gossip
       Node2、Node3得知

       Node2、Node3
         ↓ Gossip
       Node4、Node5、Node6得知

       ... 信息逐渐传播到所有节点
```

#### Gossip消息类型

| 消息类型 | 作用 | 触发时机 |
|---------|------|---------|
| MEET | 新节点加入集群 | CLUSTER MEET命令 |
| PING | 主动探测节点存活 | 定期（每秒随机选节点） |
| PONG | 响应探测 | 收到PING后 |
| FAIL | 节点故障通知 | 主观下线时 |
| FAILOVER | 故障转移通知 | 从节点升级时 |
| UPDATE | 槽分配更新 | 槽迁移完成时 |

#### 为什么Gossip比中心化更好？

```
中心化方案（不好）：
- 中心节点宕机 → 整个集群通信瘫痪
- 中心节点压力大 → 可能成为瓶颈

Gossip方案（好）：
- 无中心节点 → 任何节点宕机都不影响
- 负载分散 → 每节点只与少数节点通信
```

#### 实现要点

```cpp
class GossipProtocol {
private:
    // Gossip消息格式
    struct GossipMessage {
        MessageType type;
        std::string sender_id;
        uint64_t cluster_epoch;
        std::vector<NodeInfo> nodes;
    };

    struct NodeInfo {
        std::string node_id;
        std::string ip;
        int port;
        NodeRole role;
        uint16_t slot_start, slot_end;
        uint64_t last_pong_time;
        uint32_t failure_count;
    };

public:
    // 定期发送Gossip消息
    void run_gossip() {
        // 每秒随机选择5个节点发送PING
        auto targets = select_random_nodes(5);
        for (auto& target : targets) {
            send_ping(target);
        }
    }

    // 处理接收到的Gossip消息
    void handle_message(const GossipMessage& msg) {
        switch (msg.type) {
            case PING: handle_ping(msg); break;
            case PONG: handle_pong(msg); break;
            case FAIL: handle_fail(msg); break;
            case UPDATE: handle_update(msg); break;
        }
    }
};
```

#### 验收标准

- Gossip消息正确发送和接收
- 节点状态能正确同步

### 5.3.3 节点发现

#### 为什么需要节点发现？

节点发现让新节点能够自动融入集群：

| 需求 | 说明 |
|-----|------|
| **引导机制** | 新节点只需要知道集群中任意一个节点 |
| **自动传播** | 通过Gossip协议，信息自动传播到所有节点 |
| **槽分配同步** | 新节点加入后能获取当前的槽分配信息 |

#### 节点加入流程

```
新节点启动时：
1. 配置文件或参数指定已知节点IP:Port
2. 新节点发送CLUSTER MEET到已知节点
3. 已知节点将新节点加入自己的节点列表
4. 已知节点通过Gossip广播新节点信息
5. 所有节点陆续得知新节点存在
6. 新节点通过Gossip学习到完整的集群拓扑
```

#### CLUSTER MEET命令

```
# 新节点执行：
CLUSTER MEET 192.168.1.10 6379

# 相当于告诉集群：
"请把我介绍给集群，我是新成员"
```

#### 为什么只需要知道一个节点？

```
不需要知道所有节点，因为：
- 通过Gossip协议，一个节点知道的信息会传播给其他节点
- 第一个节点会告诉第二个，第二个会告诉第三个...
- 最终所有节点都知道新节点的存在
```

#### 验收标准

- 新节点能正确加入集群
- 所有节点都能发现新节点

### 5.4.2 请求路由

#### 为什么需要请求路由？

| 需求 | 说明 |
|-----|------|
| **本地槽处理** | 本节点负责的槽，直接处理 |
| **MOVED重定向** | 槽已迁移，客户端更新映射 |
| **ASK重定向** | 槽迁移中，临时转向目标节点 |
| **客户端缓存** | 客户端缓存槽映射减少重定向 |

#### MOVED vs ASK重定向

| 类型 | 原因 | 客户端行为 | 后续请求 |
|-----|------|-----------|---------|
| MOVED | 槽永久迁移到新节点 | 更新本地槽映射 | 直接请求新节点 |
| ASK | 槽正在迁移中（迁移还没完成） | 临时转向新节点 | 继续请求源节点 |

#### 为什么需要两种重定向？

```
MOVED：槽已经迁移完成
- 告诉客户端槽的新位置
- 客户端更新缓存，下次直接访问新节点

ASK：槽正在迁移中（迁移还没完成）
- 源节点还有数据，目标节点也有部分数据
- 如果客户端收到ASK后直接访问目标节点
- 源节点数据会丢失
- 所以ASK只做临时转向，不更新客户端映射
```

#### 请求路由流程

```
客户端发送 GET user:1000
    ↓
Node1收到请求
    ↓
计算槽号：CRC16("user:1000") % 16384 = 12345
    ↓
Node1查槽12345归属 → Node2
    ↓
12345在Node1本地？ → 否
    ↓
返回 MOVED 12345 192.168.1.20:6379
    ↓
客户端收到MOVED，更新映射
    ↓
下次直接请求Node2
```

#### 响应格式

```
MOVED <slot> <ip>:<port>    # 槽已迁移，客户端应更新映射
ASK <slot> <ip>:<port>      # 槽迁移中，客户端应临时转向
```

#### 实现要点

```cpp
class RequestRouter {
private:
    SlotMap* slot_map_;
    Cluster* cluster_;

public:
    Response route(const std::string& key, const Command& cmd) {
        uint16_t slot = SlotMap::key_to_slot(key);
        std::string owner = slot_map_->get_slot_owner(slot);

        if (owner == current_node_id_) {
            // 本地槽，直接处理
            return execute_local(key, cmd);
        }

        // 检查槽迁移状态
        if (slot_map_->is_migrating(slot)) {
            return ask_redirect(slot, owner);
        }

        // 返回MOVED重定向
        return moved_redirect(slot, owner);
    }
};
```

#### 验收标准

- 本地槽请求正确处理
- 非本地槽返回正确的MOVED/ASK响应

### 5.4.3 槽迁移

#### 为什么需要槽迁移？

槽迁移是集群扩缩容的核心：

| 需求 | 说明 |
|-----|------|
| **动态扩缩容** | 添加/移除节点时重新分配槽 |
| **在线迁移** | 迁移过程中服务不中断 |
| **数据完整性** | 确保迁移过程中数据不丢失 |
| **状态透明** | 客户端无感知 |

#### 槽迁移流程

```
场景：Node1的槽12345需要迁移到Node2

阶段1：开始迁移
- Node1执行 CLUSTER SETSLOT 12345 MIGRATING Node2
- Node1标记槽12345为迁移中状态
- Node2执行 CLUSTER SETSLOT 12345 IMPORTING Node1
- Node2标记槽12345为迁入状态

阶段2：数据迁移
- Node1遍历槽12345的所有键
- 将每个键值对发送到Node2
- Node2接收并存储这些数据

阶段3：完成迁移
- Node1确认所有数据已迁移
- 执行 CLUSTER SETSLOT 12345 Node2
- 所有节点更新槽映射，Node1不再负责槽12345
```

#### 迁移中的请求处理

```
客户端请求槽12345的数据：

情况1：客户端请求到Node1（旧节点）
- Node1发现槽12345正在迁移
- 返回 ASK 12345 Node2:6379
- 客户端临时转向Node2

情况2：客户端请求到Node2（新节点）
- Node2发现槽12345是迁入状态
- 处理请求
- 如果数据不存在，向Node1获取

情况3：迁移完成后
- 槽12345已完全属于Node2
- 返回 MOVED 12345 Node2:6379
```

#### 为什么迁移过程不会丢失数据？

```
关键：ASK只做临时转向，不更新客户端映射

客户端访问Node1（迁移中）：
1. Node1检查数据是否已迁移 → 否
2. 返回ASK，指向Node2
3. 客户端转向Node2获取数据
4. Node2没有数据时，向Node1请求

这样确保：
- 迁移过程中的请求都能正确处理
- 客户端映射不会在迁移中途更新
- 数据完整性得到保证
```

#### 实现要点

```cpp
class SlotMigration {
private:
    uint16_t slot_;
    std::string source_node_;
    std::string target_node_;
    MigrationState state_;  // START/IN_PROGRESS/COMPLETE

public:
    void start_migration(uint16_t slot, const std::string& target) {
        slot_ = slot;
        target_node_ = target;
        state_ = MIGRATING;

        // 通知源节点开始迁出
        send_to_source(SETSLOT MIGRATING);
        // 通知目标节点开始迁入
        send_to_target(SETSLOT IMPORTING);
    }

    void migrate_keys() {
        // 遍历源节点上该槽的所有键
        auto keys = source_node_->get_keys_in_slot(slot_);
        for (auto& key : keys) {
            auto value = source_node_->get(key);
            target_node_->import(key, value);
            source_node_->delete(key);
        }
    }

    void complete_migration() {
        state_ = COMPLETE;
        // 广播槽归属变更
        broadcast_slot_update(slot_, target_node_);
    }
};
```

#### 验收标准

- 槽迁移能正确完成
- 迁移过程中数据不丢失
- 迁移过程中请求能正确处理

### 5.4.4 动态扩缩容

#### 为什么需要动态扩缩容？

分布式系统需要能够动态调整容量：

| 需求 | 说明 |
|-----|------|
| **水平扩展** | 通过添加节点增加容量 |
| **缩容支持** | 节点移除时自动迁移数据 |
| **负载均衡** | 槽均匀重新分配 |
| **业务无感知** | 扩缩容过程服务不中断 |

#### 扩容场景示例

```
3节点集群扩容到4节点：

扩容前（3节点）：
Node1: 槽0-5460
Node2: 槽5461-10922
Node3: 槽10923-16383

扩容步骤：
1. 新节点Node4加入集群（此时没有槽）
2. 从各主节点迁移一部分槽到Node4
3. 最终每个节点约4096个槽

扩容后（4节点）：
Node1: 槽0-4095
Node2: 槽4096-8191
Node3: 槽8192-12287
Node4: 槽12288-16383
```

#### 扩缩容对业务的影响

```
扩缩容前：
3节点集群，QPS 10万

扩缩容中：
- 槽迁移是后台进行的
- 正常请求通过MOVED/ASK重定向处理
- 对业务有一定影响（约10-20%性能波动）

扩缩容后：
4节点集群，QPS 13万（理论线性扩展）
```

#### 验收标准

- 新节点能加入并分配槽
- 节点移除后槽能重新分配
- 扩缩容过程不影响服务

## 5.5 章节4：主从复制

### 5.5.1 问题分析

```
单机版的单点故障问题：
┌─────────────────────────────────────────────────────────────┐
│  ┌─────────────────────┐                                   │
│  │    主节点           │                                   │
│  │  ┌───────────────┐  │                                   │
│  │  │ 所有数据      │  │                                   │
│  │  └───────────────┘  │                                   │
│  └─────────────────────┘                                   │
│                                                              │
│  问题：                                                    │
│  - 节点挂了 → 数据丢失                                    │
│  - 无法继续服务                                           │
└─────────────────────────────────────────────────────────────┘

主从复制的解决方案：
┌─────────────────────────────────────────────────────────────┐
│  ┌─────────────────────┐     ┌─────────────────────┐       │
│  │    主节点           │ ←── │    从节点           │       │

└─────────────────────────────────────────────────────────────┘
```

#### 主从复制的价值

| 特性 | 说明 |
|-----|------|
| **数据冗余** | 从节点保存主节点的数据副本 |
| **读写分离** | 从节点可提供读服务，分担主节点压力 |
| **故障恢复** | 主节点故障时从节点可升级 |
| **异地灾备** | 从节点可部署在不同机房 |

### 5.5.2 全量复制

#### 为什么需要全量复制？

从节点刚加入或复制链路断开时，需要全量复制来同步数据：

| 场景 | 说明 |
|-----|------|
| **数据初始化** | 从节点第一次同步，需要完整数据 |
| **断连恢复** | 复制链路断开后的重新同步 |
| **保证一致性** | 确保从节点数据与主节点完全一致 |

#### 全量复制流程

```
从节点视角：
1. 发送 PSYNC ? -1
   （？表示不知道主节点runid，-1表示全量复制）

主节点视角：
2. 返回 +FULLRESYNC <runid> <offset>
   （告诉从节点全量复制，并用runid和offset标识）

3. 执行BGSAVE生成RDB文件
   （fork子进程，父进程继续处理请求）

4. RDB文件生成后，发送给从节点
   （可能很大，需要分片传输）

5. 发送RDB文件期间的增量命令
   （主节点将这些命令存入复制缓冲区）

从节点视角：
6. 接收RDB文件并加载到内存
7. 接收增量命令并重放
8. 复制开始，正常接收增量命令
```

#### PSYNC命令解析

```
PSYNC <runid> <offset>

runid：
- ? 表示第一次同步，需要全量复制
- 主节点runid 表示增量复制（部分同步）

offset：
- -1 表示全量复制
- 正整数表示从该偏移量开始增量复制
```

#### 验收标准

- 全量复制正常工作
- 数据完整同步

---

### 5.5.3 增量复制

#### 为什么需要增量复制？

全量复制代价很高（需要生成和传输RDB文件），增量复制让正常情况下的同步更高效：

| 特性 | 说明 |
|-----|------|
| **低开销** | 只传输增量命令，无需传输整个数据集 |
| **高效率** | 网络传输量小，同步延迟低 |
| **高吞吐** | 主节点可以支持更多从节点 |
| **积压缓冲区** | 记录近期命令，支持断线重连 |

#### 增量复制工作原理

```
主节点：
1. 每个写命令执行时，同步写入两个地方：
   - 客户端返回（正常处理）
   - 复制积压缓冲区（环形缓冲区）

2. 定期PING从节点，携带当前复制偏移量

从节点：
3. 定期发送REPLCONF ACK <offset>
   （告诉主节点，我已经同步到哪个位置了）

4. 如果从节点偏移量落后，主节点补发缺失的命令
```

#### 复制积压缓冲区

```
环形缓冲区结构（默认1MB）：

┌────────────────────────────────────────────────────────┐
│ cmd1 │ cmd2 │ cmd3 │ cmd4 │ cmd5 │ ... │ cmdN │     │
└────────────────────────────────────────────────────────┘
 ↑写指针                                      ↑读指针

- 新命令写入：写指针向前移动
- 从节点确认：读指针向前移动
- 缓冲区满：覆盖最旧的数据

如果从节点断开重连：
- 从节点发送 REPLCONF ACK <offset>
- 主节点检查从节点偏移量
- 如果在积压缓冲区范围内 → 只发送缺失的命令
- 如果不在范围内 → 需要全量复制
```

#### 验收标准

- 增量复制正常工作
- 主从数据最终一致

---

### 5.5.4 只读从节点

#### 为什么需要只读？

从节点默认只读，既保证了数据一致性，又提高了系统可用性：

| 特性 | 说明 |
|-----|------|
| **数据一致性** | 防止从节点被意外写入导致数据不一致 |
| **读写分离** | 读请求分流到从节点，减轻主节点压力 |
| **故障转移** | 从节点可升级为主节点，不应包含写操作 |
| **安全防护** | 防止应用程序误写入从节点 |

#### 只读模式的行为

```
客户端尝试在从节点执行写命令：
127.0.0.1:6380> SET key value
(error) READONLY You can't write against a read only replica.

只读从节点正确响应读命令：
127.0.0.1:6380> GET key
"value"
127.0.0.1:6380> EXISTS key
(integer) 1
```

#### 验收标准

- 从节点能处理读命令
- 从节点拒绝写命令

### 5.6.2 故障检测

#### 为什么需要故障检测？

故障检测是故障转移的前提：

| 机制 | 说明 |
|-----|------|
| **心跳机制** | 通过PING/PONG检测节点存活 |
| **主观下线** | 单个节点认为另一个节点故障 |
| **客观下线** | 集群认为某个节点真正故障 |
| **故障传播** | 故障信息广播到所有节点 |

#### 为什么需要主观+客观两层检测？

```
主观下线（单节点判断）：
- 节点A在15秒内没收到节点B的PONG
- 节点A认为节点B故障
- 但这可能只是网络抖动或节点B负载高

客观下线（集群判断）：
- 超过半数的节点都认为节点B故障
- 才认定节点B真正下线
- 避免误判导致的频繁故障转移
```

#### 心跳检测流程

```
每秒执行：
1. 随机选择5个节点发送PING
2. 如果节点在cluster-node-timeout内没回复PONG
3. failure_count++
4. 如果failure_count超过阈值（默认3）
5. 该节点标记为疑似下线
```

#### 验收标准

- 故障节点能被正确检测
- 主观下线和客观下线正确判断

---

### 5.6.3 从节点选举

#### 为什么需要选举？

主节点故障后，需要从节点通过选举机制升级为新主节点：

| 特性 | 说明 |
|-----|------|
| **Raft风格选举** | 从节点发起选举，获得多数票当选 |
| **数据最新优先** | 复制偏移量最新的从节点优先 |
| **epoch机制** | 防止重复投票和无效选举 |
| **自动进行** | 故障转移自动触发，不需要人工干预 |

#### 选举条件

```
不是所有从节点都能发起选举，必须满足：

1. 从节点的主节点处于客观下线状态
   （确保主节点真的故障了）

2. 从节点复制偏移量足够新
   （数据不能比主节点落后太多）

3. 从节点未执行其他故障转移
   （避免多个从节点同时选举）
```

#### Raft风格选举流程

```
从节点C视角（主节点A的从节点）：

1. 等待A客观下线
   （确认主节点真的故障了）

2. 增加currentEpoch（选举轮次）
   currentEpoch = A的currentEpoch + 1

3. 发送FAILOVER_AUTH_REQUEST给所有主节点
   消息包含：
   - 我的currentEpoch
   - 我的复制偏移量
   - 我主节点的runid

4. 主节点收到后：
   - 检查是否投过票（每个epoch只能投一次）
   - 检查从节点数据是否够新
   - 满足条件则返回FAILOVER_AUTH_ACK

5. 从节点等待多数主节点的投票
   多数 = 主节点总数/2 + 1
   例如：3个主节点 → 需要2票

6. 获得多数票后：
   - 成为新主节点
   - 开始接收写请求
   - 向集群广播PONG消息
```

#### 为什么需要epoch？

```
避免重复投票和无效选举：

旧epoch（可能来自网络分区）：
- 节点认为A故障，发送投票请求
- 但A实际上还活着，只是网络慢

新epoch：
- A恢复后有新epoch
- 旧epoch的投票无效
- 确保选举结果是最新的
```

#### 验收标准

- 从节点能正确发起选举
- 选举过程正确，多数票生效

---

### 5.6.4 故障转移执行

#### 为什么需要故障转移执行？

故障转移执行是将集群从"有节点故障"状态恢复到"正常"状态的关键步骤：

| 步骤 | 说明 |
|-----|------|
| **从变主** | 从节点升级为新的主节点 |
| **槽接管** | 新主节点接管原主节点的槽 |
| **拓扑更新** | 所有节点更新集群拓扑 |
| **旧主处理** | 原主节点恢复后变成从节点 |

#### 故障转移执行流程

```
场景：主节点A故障，从节点C当选新主节点

步骤1：C成为新主节点
- C向所有节点广播PONG消息
- 消息包含：A的槽现在归C管

步骤2：其他节点更新拓扑
- 所有节点收到C的PONG
- 更新槽映射：A的槽 → C
- 更新节点信息：C角色：MASTER

步骤3：C开始接收请求
- C现在负责A的所有槽
- 可以接收客户端的读写请求

步骤4：A恢复后处理
- A重新上线，发送PING
- 其他节点告诉A：C现在是主节点
- A变成C的从节点，开始复制C的数据
```

#### 故障转移前后对比

```
故障转移前：
┌───────┐         ┌───────┐
│  A(M) │ ──复制─→ │  C(R) │
│槽0-500│         │ 数据副本│
└───────┘         └───────┘

故障转移后：
┌───────┐         ┌───────┐
│  A(R) │ ←─复制── │  C(M) │
│       │          │槽0-500│
└───────┘          └───────┘
  变成从节点        升级为主节点
```

#### 验收标准

- 故障转移后新主节点正常工作
- 槽接管正确
- 数据不丢失
