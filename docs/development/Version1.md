# 第二章：骨架版本（Version1）

## 版本概述

**版本目标**：构建项目最小可运行版本，让整个系统能够跑起来。

**问题分析**：作为一个学习项目，你从来没有实现过一个完整的服务器程序。如果一开始就做多线程、内存池、分布式集群：
- 代码复杂度爆炸，难以调试
- 问题难以定位，不知道哪里出错
- 无法理解核心组件之间的协作关系

**为什么需要骨架版本**：
- 先解决"能不能跑"的问题
- 建立对整个系统的直觉
- 验证核心设计决策是否正确
- 为后续迭代打好基础

**核心功能**：
- 单Reactor网络模型
- RESP协议解析
- 命令模式与命令工厂
- 全局键值存储
- GET/SET/DEL/EXISTS命令

**预计开发周期**：2-3周

**预计输出**：约800-1000行代码

### 2.1.2 请求处理流程

```
一个 SET 命令的完整旅程：

1. 客户端发送 "SET name zhangsan\r\n"
   │
   ▼
2. TCP 数据到达服务器的 socket
   │
   ▼
3. epoll 检测到 socket 可读
   │
   ▼
4. EventLoop 回调 Connection::handle_read()
   │
   ▼
5. Connection 从 socket 读取数据到 Buffer
   │
   ▼
6. RespParser 从 Buffer 解析出命令
   │
   ▼
7. CommandFactory 根据命令名创建具体命令对象
   │
   ▼
8. 命令对象执行：从 GlobalStorage 读写数据
   │
   ▼
9. RespEncoder 将执行结果编码成响应
   │
   ▼
10. Connection 将响应写入 Buffer
    │
    ▼
11. epoll 检测到 socket 可写
    │
    ▼
12. EventLoop 回调 Connection::handle_write()
    │
    ▼
13. Connection 将响应发送给客户端
```

## 2.2 网络层详解

### 2.2.1 为什么需要 EventLoop？

**问题**：服务器需要同时处理多个客户端连接

**解决方案A：多线程（不适合 IO 密集型）**
```
每个客户端一个线程：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 线程1   │ │ 线程2   │ │ 线程3   │   ← 每个线程都要占用栈内存（1MB+）
│ accept  │ │ accept  │ │ accept  │
│ read    │ │ read    │ │ read    │
│ write   │ │ write   │ │ write   │
└─────────┘ └─────────┘ └─────────┘
问题：
- 线程创建/销毁开销大
- 上下文切换开销大
- 大部分时间在等待 I/O，CPU 闲置
```

**解决方案B：EventLoop + epoll（适合 IO 密集型）**
```
单线程事件循环：
┌─────────────────────────────────────┐
│           EventLoop                  │
│                                     │
│   epoll_wait() 阻塞等待              │
│   ────────────────────────          │
│   有事件才处理，无事件就等待          │
│                                     │
│   处理连接1 ──► 处理连接2 ──► ...   │
└─────────────────────────────────────┘

优势：
- 不用创建/销毁线程
- 不用频繁上下文切换
- CPU 始终在干活（处理实际事件）
```

### 2.2.2 epoll 工作原理

**为什么选择 epoll 而不是 select/poll？**

| 方案 | 最大 fd 限制 | 时间复杂度 | 问题 |
|-----|-------------|-----------|------|
| select | 1024 | O(n) | 需要遍历所有 fd |
| poll | 无限制 | O(n) | 需要遍历所有 fd |
| epoll | 无限制 | O(1) | 只需要处理活跃的 fd |

**epoll 三部曲**：
```cpp
// 1. 创建 epoll 实例
int epfd = epoll_create(1);

// 2. 注册要监听的事件
struct epoll_event ev;
ev.events = EPOLLIN;  // 监听可读事件
ev.data.fd = listen_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listen_fd, &ev);

// 3. 等待事件发生
struct epoll_event events[MAX_EVENTS];
int nfds = epoll_wait(epfd, events, MAX_EVENTS, -1);
for (int i = 0; i < nfds; i++) {
    // 处理 events[i] 对应的 fd
}
```

### 2.2.3 Channel、Connection、Socket 的职责划分

**为什么需要三个类而不是一个？**

```
单一职责原则：
┌─────────────────────────────────────────────────────────────┐
│  Socket                                                    │  │
│  - 封装 socket 文件描述符                                 │  │
│  - 提供 bind/listen/accept/connect 等底层操作            │  │
│  - 不关心业务逻辑                                         │  │
├─────────────────────────────────────────────────────────────┤
│  Channel                                                   │  │
│  - 封装 epoll 事件                                        │  │
│  - 管理 fd 与 event callback 的映射                       │  │
│  - 处理可读/可写/错误等事件                               │  │
│  - 不关心 socket 的具体行为                               │  │
├─────────────────────────────────────────────────────────────┤
│  Connection                                                │  │
│  - 管理一个客户端连接的状态                                │  │
│  - 包含 socket、buffer、parser                           │  │
│  - 处理具体的业务逻辑（读请求、写响应）                    │  │
└─────────────────────────────────────────────────────────────┘

好处：
- Socket 可以被复用（不只是 TCP 连接）
- Channel 可以独立测试
- Connection 业务逻辑清晰
```

### 2.2.4 粘包问题与 Buffer 设计

**问题**：TCP 是流协议，数据没有边界

```
发送："SET name zhangsan\r\n"
接收：可能收到：
  - "SET n"
  - "SET name zhangsan\r\n"
  - "SET name zhangsan\r\nGET key\r\n"
  - 任何上述情况的组合
```

**解决方案：换行符分割 + 缓冲区**

```cpp
// Buffer 双指针设计
class Buffer {
private:
    std::vector<char> buffer_;  // 底层存储
    size_t read_index_ = 0;    // 读指针
    size_t write_index_ = 0;   // 写指针

public:
    // 写入数据
    void append(const char* data, size_t len) {
        // 确保有足够空间
        if (writable() < len) {
            // 可能需要扩容或移动数据
        }
        std::copy(data, data + len, &buffer_[write_index_]);
        write_index_ += len;
    }

    // 按行读取（找到 \r\n）
    std::string read_line() {
        // 找到 \r\n 的位置
        auto pos = find('\r\n');
        if (pos == npos) return "";
        // 提取数据
        std::string line(buffer_.begin() + read_index_,
                         buffer_.begin() + pos);
        read_index_ = pos + 2;  // 跳过 \r\n
        return line;
    }
};
```

## 2.3 协议层详解

### 2.3.1 为什么需要 RESP 协议？

**问题**：TCP 发送的是字节流，服务器怎么知道客户端发送了多少数据？

**解决方案**：定义应用层协议，让数据有明确的边界和格式

**RESP（Redis Serialization Protocol）协议特点**：
- 简单易解析
- 二进制安全
- 人类可读（调试方便）

### 2.3.2 RESP 协议格式

```
┌─────────────────────────────────────────────────────────────┐
│                      RESP 数据类型                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  简单字符串（Simple String）：                               │
│  +OK\r\n                                                    │
│  第一个字节是 '+'                                           │
│                                                             │
│  错误（Error）：                                            │
│  -ERR unknown command 'getk'\r\n                           │
│  第一个字节是 '-'                                           │
│                                                             │
│  整数（Integer）：                                          │
│  :100\r\n                                                  │
│  第一个字节是 ':'                                           │
│                                                             │
│  批量字符串（Bulk String）：                                  │
│  $5\r\nhello\r\n                                           │
│  $ 表示后续是字符串长度，然后是 \r\n，然后是实际数据         │
│                                                             │
│  数组（Array）：                                            │
│  *3\r\n                                                    │
│  :1\r\n                                                    │
│  :2\r\n                                                    │
│  :3\r\n                                                    │
│  *3 表示数组有3个元素，后续是3个元素的数据                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.3.3 RESP 解析器工作原理

```
解析 "SET name zhangsan\r\n"：

原始数据：
*3\r\n$3\r\nSET\r\n$4\r\nname\r\n$8\r\nzhangsan\r\n

解析过程：
1. 第一个字节是 '*' → 这是一个数组
2. 读取数字 3 → 数组有3个元素
3. 读取第一个元素：'$' → 批量字符串
4. 读取长度 3 → 字符串长度是3
5. 读取 "SET"
6. 读取第二个元素：'$' → 批量字符串
7. 读取长度 4 → 字符串长度是4
8. 读取 "name"
9. 读取第三个元素：'$' → 批量字符串
10. 读取长度 8 → 字符串长度是8
11. 读取 "zhangsan"

解析结果：
RespValue {
    type = ARRAY,
    array = [
        RespValue { type = BULK_STRING, data = "SET" },
        RespValue { type = BULK_STRING, data = "name" },
        RespValue { type = BULK_STRING, data = "zhangsan" }
    ]
}
```

## 2.4 应用层详解

### 2.4.1 为什么需要命令模式？

**问题**：需要处理多种不同的命令（GET、SET、DEL、EXISTS...）

**命令模式的优势**：

```
不用命令模式（问题）：
┌─────────────────────────────────────────────────────────────┐
│  void handle_command(const vector<string>& args) {          │
│      if (args[0] == "GET") {                              │
│          // 100行 GET 处理逻辑                              │
│      } else if (args[0] == "SET") {                       │
│          // 150行 SET 处理逻辑                              │
│      } else if (args[0] == "DEL") {                       │
│          // 80行 DEL 处理逻辑                               │
│      }                                                     │
│      // ... 更多 else if ...                               │
│  }                                                        │
│                                                             │
│  问题：                                                    │
│  - 所有逻辑混在一个函数                                    │
│  - 添加新命令要修改这个函数                                │
│  - 违反开闭原则                                            │
└─────────────────────────────────────────────────────────────┘

用命令模式（解决方案）：
┌─────────────────────────────────────────────────────────────┐
│  // 命令抽象基类                                            │
│  class Command {                                           │
│  public:                                                   │
│      virtual ~Command() = default;                         │
│      virtual RespValue execute(Storage* storage) = 0;     │
│  };                                                        │
│                                                             │
│  // 具体命令实现                                            │
│  class GetCommand : public Command {                       │
│  public:                                                   │
│      RespValue execute(Storage* storage) override {        │
│          // 只处理 GET 逻辑                                 │
│      }                                                     │
│  };                                                        │
│                                                             │
│  // 命令工厂                                                │
│  class CommandFactory {                                    │
│  public:                                                   │
│      static Command* create(const string& name) {         │
│          if (name == "GET") return new GetCommand();       │
│          if (name == "SET") return new SetCommand();       │
│          // ...                                            │
│      }                                                     │
│  };                                                        │
│                                                             │
│  // 使用                                                    │
│  void handle_command(const vector<string>& args) {        │
│      auto cmd = CommandFactory::create(args[0]);          │
│      cmd->execute(storage);                                │
│      delete cmd;                                           │
│  }                                                         │
│                                                             │
│  优势：                                                    │
│  - 每个命令逻辑独立                                         │
│  - 添加新命令只需添加新类                                   │
│  - 易于测试和维护                                          │
└─────────────────────────────────────────────────────────────┘
```

### 2.4.2 命令执行流程

```
SET name zhangsan 的执行流程：

1. RespParser 解析出数组 ["SET", "name", "zhangsan"]
   │
   ▼
2. CommandFactory::create("SET") 创建 SetCommand 对象
   │
   ▼
3. SetCommand::execute(storage) 被调用
   │
   ▼
4. SetCommand 内部：
   a. 验证参数数量（SET 需要 3 个参数）
   b. 调用 storage->set("name", "zhangsan")
   c. 返回 RespValue::ok()
   │
   ▼
5. RespEncoder 将结果编码为 "+OK\r\n"
   │
   ▼
6. Connection::send_response() 发送给客户端
```

## 2.5 存储层详解

### 2.5.1 为什么需要全局单例存储？

**问题**：所有命令都需要访问同一个数据存储

**解决方案**：全局单例

```
单例模式：
┌─────────────────────────────────────────────────────────────┐
│  class GlobalStorage {                                     │
│  private:                                                  │
│      static GlobalStorage* instance_;  // 唯一实例          │
│      std::unordered_map<std::string, std::string> data_;  │
│      std::shared_mutex mutex_;  // 保护 data_              │
│                                                             │
│  public:                                                   │
│      static GlobalStorage* get_instance() {                │
│          if (!instance_) {                                 │
│              instance_ = new GlobalStorage();              │
│          }                                                 │
│          return instance_;                                 │
│      }                                                     │
│                                                             │
│      void set(const string& k, const string& v) {         │
│          std::write_lock lock(mutex_);                    │
│          data_[k] = v;                                    │
│      }                                                     │
│                                                             │
│      std::string get(const string& k) {                   │
│          std::read_lock lock(mutex_);                      │
│          auto it = data_.find(k);                         │
│          return (it != data_.end()) ? it->second : "";   │
│      }                                                     │
│  };                                                        │
└─────────────────────────────────────────────────────────────┘
```

### 2.5.2 为什么要用 shared_mutex？

**问题**：读操作远远多于写操作

**解决方案**：读写锁

```
普通互斥锁的问题：
┌─────────────────────────────────────────────────────────────┐
│  mutex_.lock();                                            │
│  // 读数据（可能需要很长时间）                              │
│  read_data();                                              │
│  mutex_.unlock();                                          │
│                                                             │
│  问题：读操作期间，写操作无法进行                           │
│       即使有 100 个并发读，也只能串行                       │
└─────────────────────────────────────────────────────────────┘

读写锁的优势：
┌─────────────────────────────────────────────────────────────┐
│  // 读操作（可以多个同时读）                                │
│  read_lock.lock();                                         │
│  read_data();                                              │
│  read_lock.unlock();                                      │
│                                                             │
│  // 写操作（独占）                                          │
│  write_lock.lock();                                        │
│  write_data();                                             │
│  write_lock.unlock();                                      │
│                                                             │
│  效果：                                                    │
│  - 多个读可以并发                                           │
│  - 写操作独占                                               │
└─────────────────────────────────────────────────────────────┘

对于缓存场景，读 >> 写，读写锁能极大提升并发性能。
```

## 2.6 章节总结

### 2.6.1 骨架版本学到了什么

```
核心组件协作关系：

┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  EventLoop  │────▶│ Connection │────▶│   Buffer   │

                           │

                    ┌─────────────┐     ┌─────────────┐
```

### 2.6.2 骨架版本的局限性

```
┌─────────────────────────────────────────────────────────────┐
│                    骨架版本的不足                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 单Reactor只能利用一个CPU核心                             │
│     → 处理能力有限，无法充分发挥多核CPU性能                   │
│                                                             │
│  2. 内存分配使用malloc/free                                 │
│     → 高并发下性能差，内存碎片多                            │
│                                                             │
│  3. 全局锁保护哈希表                                        │
│     → 多线程并发读写时锁竞争严重                            │
│                                                             │
│  4. 没有过期淘汰机制                                        │
│     → 内存会无限增长                                        │
│                                                             │
│  5. 只支持String类型                                        │
│     → 无法满足复杂业务场景                                  │
│                                                             │
│  6. 没有持久化                                              │
│     → 进程重启数据全部丢失                                  │
│                                                             │
│  这些问题将在 Version2 中解决！                              │
└─────────────────────────────────────────────────────────────┘
```
