# 第四章：增强版本（Version3）

## 版本概述

**版本目标**：添加更多数据类型和持久化功能，并进行全面的性能优化。

**问题分析**：基础版本解决了"跑得快"的问题，但功能完备性不足：

```
基础版本的问题分析：
┌─────────────────────────────────────────────────────────────┐
│  问题1：只支持String类型                                    │
│  ─────────────────────────────────────────────────         │
│  所有数据都是简单的 key-value 字符串                        │
│  → 无法存储结构化数据（如用户信息、配置项）                 │
│  → 无法实现列表、集合等高级数据结构                        │
│  → 无法满足复杂业务场景                                    │
│                                                              │
│  问题2：没有持久化                                          │
│  ─────────────────────────────────────────────────         │
│  数据只存在内存中                                          │
│  → 进程重启数据全部丢失                                   │
│  → 断电数据全部丢失                                       │
│  → 无法作为真正的"数据库"使用                             │
│                                                              │
│  问题3：只有一个淘汰算法（LRU）                            │
│  ─────────────────────────────────────────────────         │
│  所有场景都使用LRU                                        │
│  → 有些场景FIFO更合适（如临时缓存）                       │
│  → 有些场景LFU更合适（如热点数据）                        │
│  → 有些场景Random就够了（如负载均衡）                      │
└─────────────────────────────────────────────────────────────┘
```

**为什么需要增强版本**：

| 需求 | 说明 |
|-----|------|
| **更多数据类型** | Hash/List/Set/ZSet满足复杂业务场景 |
| **持久化机制** | RDB + AOF 保证数据安全 |
| **多种淘汰策略** | LFU/FIFO/Random 适应不同场景 |
| **性能优化** | 锁优化、I/O优化、CPU优化 |

**预计开发周期**：8-11周

### 4.2.2 哈希数据类型

#### 为什么需要哈希？

哈希数据类型（Hash）是Redis中最常用的数据结构之一，它解决了以下问题：

| 特性 | 说明 |
|-----|------|
| **结构化存储** | 将一个对象的多个属性存储在一个键下 |
| **减少键数量** | 100个用户的用户名信息，用String需要100个键，用Hash只需要1个 |
| **原子操作** | 可以单独更新某个field，而不影响其他fields |
| **内存效率** | 相比为每个属性存储单独的键，Hash更节省内存 |

#### 使用场景示例

```
用户信息存储：
user:1000 → {name: "张三", age: 25, city: "北京"}

商品详情：
product:SKU123 → {name: "iPhone15", price: 6999, stock: 100}

配置项存储：
config:redis → {maxmemory: "4gb", port: 6379}
```

#### 命令列表

```
HSET key field value          # 设置哈希字段
HGET key field                # 获取哈希字段值
HDEL key field [field ...]    # 删除哈希字段
HEXISTS key field             # 检查字段是否存在
HGETALL key                   # 获取所有字段和值
HKEYS key                     # 获取所有字段名
HVALS key                     # 获取所有值
HLEN key                      # 获取字段数量
HINCRBY key field increment   # 字段值增加
HSETNX key field value        # 字段不存在时设置
```

#### 实现要点

```cpp
class Hash {
private:
    std::unordered_map<std::string, std::string> fields_;  // field → value

public:
    bool set(const std::string& field, const std::string& value);
    std::string get(const std::string& field);
    bool del(const std::string& field);
    bool exists(const std::string& field);
    std::map<std::string, std::string> get_all();
    int64_t incr_by(const std::string& field, int64_t increment);
};
```

### 4.2.3 列表数据类型

#### 为什么需要列表？

列表数据类型（List）是常用的有序数据结构，它解决了以下问题：

| 特性 | 说明 |
|-----|------|
| **有序存储** | 元素按插入顺序排列，可以按索引访问 |
| **两端操作** | 支持从头部（LPUSH/LPOP）和尾部（RPUSH/RPOP）高效操作 |
| **消息队列** | 可以实现简单的消息队列系统 |
| **最新N个元素** | 快速获取最新的N条记录 |

#### 使用场景示例

```
消息队列：
LPUSH将消息放入队列，RPOP取出消息（先进先出）

最新消息列表：
LPUSH将新消息插入列表，LRANGE获取最新10条

任务队列：
LPUSH添加任务，BRPOP阻塞等待任务（生产者-消费者模式）

关注列表：
按时间顺序存储用户关注的人的动态
```

#### 为什么使用双向链表而不是数组？

```
数组实现（不好）：
┌─────────────────────────────────────────────────────────────┐
│  [0] [1] [2] [3] [4] ...                                  │
│                                                              │
│  头部插入：需要移动所有元素 → O(n)                          │
│  尾部插入：O(1)                                             │
│  按索引访问：O(1)                                           │
└─────────────────────────────────────────────────────────────┘

双向链表实现（好）：
┌─────────────────────────────────────────────────────────────┐
│  HEAD ──► [A] ──► [B] ──► [C] ──► [D] ──► TAIL           │
│                                                              │
│  头部插入：HEAD = new_node; → O(1)                         │
│  尾部插入：TAIL = new_node; → O(1)                         │
│  按索引访问：需要遍历 → O(n)                                │
│                                                              │
│  优化：对于LRANGE等范围查询，可以使用更高效的数据结构        │
└─────────────────────────────────────────────────────────────┘
```

#### 命令列表

```
LPUSH key value [value ...]   # 从左侧插入
RPUSH key value [value ...]   # 从右侧插入
LPOP key                     # 从左侧弹出
RPOP key                     # 从右侧弹出
LLEN key                     # 获取列表长度
LRANGE key start stop        # 获取范围内元素
LREM key count value         # 移除指定元素
LSET key index value         # 设置指定位置元素
LINDEX key index             # 获取指定位置元素

```

## 4.3 章节2：更多缓存替换算法

### 4.3.1 问题分析

```
LRU的局限：
┌─────────────────────────────────────────────────────────────┐
│  场景：API响应缓存                                          │
│                                                              │
│  某个API在凌晨3点被访问一次（预热），然后一整天没被访问       │
│  LRU会保留这个API的缓存，因为它是"最近使用"的               │
│                                                              │
│  但实际上，另一个API被访问了10000次，却可能被淘汰了         │
│                                                              │
│  问题：LRU只看"最近"，不看"频率"                           │
└─────────────────────────────────────────────────────────────┘

解决方案：LFU（最不经常使用）
┌─────────────────────────────────────────────────────────────┐
│  LFU会保留被访问10000次的API，淘汰只被访问1次的API         │
│  → 更适合"热点数据"场景                                   │
└─────────────────────────────────────────────────────────────┘
```

#### 为什么需要多种淘汰算法？

不同的业务场景需要不同的淘汰策略：

| 算法 | 淘汰依据 | 适用场景 |
|-----|---------|---------|
| **LRU** | 最近最少使用 | 热点数据缓存、页面缓存 |
| **LFU** | 最不经常使用 | 高频访问数据、API响应缓存 |
| **FIFO** | 先进先出 | 临时数据、会话缓存 |
| **Random** | 随机淘汰 | 负载均衡、简单缓存场景 |

### 4.3.2 LFU算法

#### 为什么需要LFU？

LFU（Least Frequently Used）算法根据数据的访问频率进行淘汰：

| 特性 | 说明 |
|-----|------|
| **高频优先保留** | 访问频率高的数据更可能是热点数据 |
| **区分冷热数据** | 能够区分访问次数差异明显的数据 |
| **适用持续热点** | 某些数据被频繁访问，即使最近没有被访问，也应该保留 |

#### 与LRU的区别

- LRU：看"最近"，如果数据刚刚被访问过就保留
- LFU：看"频率"，如果数据被访问过很多次就保留

#### 为什么需要计数器衰减？

```
问题：计数器会无限增长
┌─────────────────────────────────────────────────────────────┐
│  场景：某个数据过去被访问了10000次，但现在已不再使用         │
│                                                              │
│  如果不减：                                                  │
│  → 这个数据的计数器一直是10000                               │
│  → 永远不会被淘汰（因为频率最高）                           │
│  → 内存浪费                                                 │
└─────────────────────────────────────────────────────────────┘

解决方案：计数器衰减
→ 随着时间推移，计数器逐渐减小
→ 长时间不访问的数据，计数器会变得很小
→ 容易被淘汰
```

#### 实现要点

```cpp
class LFUEvictionPolicy : public EvictionPolicy {
private:
    struct CounterInfo {
        uint32_t counter_;          // 访问计数
        int64_t last_access_time_; // 上次访问时间
    };
    std::unordered_map<std::string, CounterInfo> infos_;

    // 衰减因子（每分钟衰减一半）
    static constexpr double DECAY_FACTOR = 0.5;

public:
    void on_access(const std::string& key) override {
        // 增加计数
        infos_[key].counter_++;
        infos_[key].last_access_time_ = now();

        // 衰减其他计数器
        decay_counters();
    }

    std::string evict() override {
        // 找到计数器最小的键
        auto it = std::min_element(infos_.begin(), infos_.end(),
            [](auto& a, auto& b) { return a.second.counter_ < b.second.counter_; });
        std::string key = it->first;
        infos_.erase(it);
        return key;
    }
};
```

#### 验收标准

- LFU算法正确工作
- 频繁访问的键不会被淘汰
- 计数器衰减机制正常

### 4.3.3 FIFO算法

#### 为什么需要FIFO？

FIFO（First In First Out）算法按照数据进入缓存的先后顺序进行淘汰：

| 特性 | 说明 |
|-----|------|
| **简单实现** | 不需要维护复杂的访问信息 |
| **公平性** | 保证每个数据都有平等的被淘汰机会 |
| **适合短期数据** | 适合存储临时数据、session数据等 |

#### 使用场景

```
临时缓存：
存储计算中间结果，结果只需要保留一轮计算

会话缓存：
用户会话数据，会话过期后按顺序淘汰

限流窗口：
存储最近的请求记录，按时间顺序清理
```

#### 实现要点

```cpp
class FIFOEvictionPolicy : public EvictionPolicy {
private:
    std::list<std::string> fifo_list_;  // 头部最新，尾部最旧
    std::unordered_map<std::string, std::list<std::string>::iterator> pos_map_;

public:
    void on_access(const std::string& key) override {
        // FIFO中访问不改变顺序（与LRU不同）
    }

    void on_set(const std::string& key) override {
        if (pos_map_.find(key) == pos_map_.end()) {
            fifo_list_.push_front(key);
            pos_map_[key] = fifo_list_.begin();
        }
    }

    std::string evict() override {
        auto key = fifo_list_.back();
        fifo_list_.pop_back();
        pos_map_.erase(key);
        return key;
    }
};
```

### 4.3.4 Random算法

#### 为什么需要Random？

Random算法随机选择一个键进行淘汰，是最简单的淘汰策略：

| 特性 | 说明 |
|-----|------|
| **零开销** | 不需要维护任何额外信息 |
| **无偏好性** | 对所有数据一视同仁 |
| **负载均衡** | 防止某些数据长期占用缓存 |

#### 使用场景

```
简单缓存场景：
不需要考虑访问模式，只需保证缓存不溢出

基准测试：
作为其他淘汰算法的基准对比

防御性缓存：
防止缓存被特定数据占满
```

#### 实现要点

```cpp
class RandomEvictionPolicy : public EvictionPolicy {
private:
    std::vector<std::string> keys_;  // 所有键的列表

public:
    void on_access(const std::string& key) override {
        // Random中访问不改变任何状态
    }

    void on_set(const std::string& key) override {
        keys_.push_back(key);
    }

    void on_del(const std::string& key) override {
        keys_.erase(std::remove(keys_.begin(), keys_.end(), key), keys_.end());
    }

    std::string evict() override {
        std::random_shuffle(keys_.begin(), keys_.end());
        auto key = keys_.back();
        keys_.pop_back();
        return key;
    }
};
```

#### 验收标准

- Random算法正确工作
- 每次淘汰随机选择

### 4.3.5 动态切换

#### 为什么需要动态切换？

动态切换允许在运行时改变缓存淘汰策略：

| 特性 | 说明 |
|-----|------|
| **灵活适配** | 不同业务场景需要不同策略 |
| **A/B测试** | 可以测试不同策略的效果 |
| **运维友好** | 线上问题可以快速调整策略 |

#### 支持的可配置策略

| 策略 | 描述 | 适用场景 |
|-----|------|---------|
| volatile-lru | 从已设置过期时间的键中淘汰LRU | 混合数据，有明确冷热区分 |
| allkeys-lru | 从所有键中淘汰LRU | 全部数据都是缓存，无明确过期 |
| volatile-lfu | 从已设置过期时间的键中淘汰LFU | 高频访问数据带过期时间 |

| volatile-fifo | 从已设置过期时间的键中淘汰FIFO | 临时数据有过期时间 |
| allkeys-fifo | 从所有键中淘汰FIFO | 全部数据都是临时的 |
| volatile-random | 从已设置过期时间的键中随机淘汰 | 无明显访问规律 |
| allkeys-random | 从所有键中随机淘汰 | 无明显访问规律 |
| noeviction | 不淘汰，返回错误 | 不允许淘汰，严格控制内存 |

## 4.4 章节3：RDB持久化

### 4.4.1 问题分析

```
内存缓存的致命问题：
┌─────────────────────────────────────────────────────────────┐
│  数据只存在内存中                                           │
│                                                              │
│  风险场景：                                                  │
│  1. 进程异常退出 → 数据丢失                                │
│  2. 服务器断电 → 数据丢失                                  │
│  3. 服务器升级 → 需要重启 → 数据丢失                        │
│  4. 内存故障（bit flip）→ 数据损坏                        │
│                                                              │
│  解决方案：持久化                                           │
│  → 把内存数据保存到磁盘                                     │
│  → 即使重启也能恢复数据                                     │
└─────────────────────────────────────────────────────────────┘
```

#### 为什么需要持久化？

| 特性 | 说明 |
|-----|------|
| **数据备份** | 定期保存数据快照，防止数据丢失 |
| **快速恢复** | 重启时直接加载RDB文件，恢复速度快 |
| **文件紧凑** | 二进制格式，文件体积小，便于传输和备份 |
| **灾备恢复** | 可以定期备份到远程服务器 |

#### RDB vs AOF对比

| 特性 | RDB | AOF |
|-----|-----|-----|
| 文件大小 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢（需要重放命令） |
| 数据安全性 | 可能丢失快照后的数据 | 可配置（always/everysec/no） |
| 写入方式 | 定时快照 | 追加写入 |
| 优先级 | 启动时先检查 | 启动时优先加载 |

### 4.4.2 RDB文件格式

```
┌─────────────────────────────────────────────────────────────┐
│                      RDB文件格式                            │
├─────────────────────────────────────────────────────────────┤
│  +------+------------------------------------------------+│
│  | MAGIC |              RDB文件标识 "REDIS0009"          ││
│  +------+------------------------------------------------+│
│  | 辅助字段 | version, aux_field等                        ││
```

```
二进制格式（好）：
```

### 4.4.3 RDB文件生成

#### SAVE vs BGSAVE

```
SAVE（同步，阻塞）：
┌─────────────────────────────────────────────────────────────┐
│  主线程执行                                                 │
│  ─────────────────────────────────────────────────         │
│  while (saving) {                                          │
│      // 阻塞等待                                           │
│  }                                                        │
│  → 保存完成前无法处理请求                                  │
└─────────────────────────────────────────────────────────────┘

BGSAVE（异步，不阻塞）：
┌─────────────────────────────────────────────────────────────┐
│  fork子进程执行                                            │
│  ─────────────────────────────────────────────────         │
│  主进程：继续处理请求                                      │
│  子进程：生成RDB文件                                      │
│  → 对请求处理无影响                                       │
└─────────────────────────────────────────────────────────────┘
```

#### 为什么用fork而不是线程？

```
fork的问题：
┌─────────────────────────────────────────────────────────────┐
│  fork后：                                                  │
│  - 子进程拷贝父进程的地址空间（Copy-On-Write）              │
│  - 如果数据量大，内存占用翻倍                             │
│  - fork操作本身可能很慢（特别是虚拟内存大）               │
└─────────────────────────────────────────────────────────────┘

为什么还要用fork？
┌─────────────────────────────────────────────────────────────┐
│  优点：                                                    │
│  - 子进程崩溃不影响主进程                                  │
│  - 天然隔离，不需要额外的进程间通信                         │
│  - Redis历史原因，兼容性考虑                              │
└─────────────────────────────────────────────────────────────┘
```

#### 实现要点

```cpp
class RDBPersistence {
private:
    std::atomic<bool> bg_save_in_progress_;
    pid_t child_pid_;

public:
    // 同步保存（主线程阻塞）
    void save() {
        // 遍历所有数据库
        // 写入每个键值对
        // 计算CRC64
        // 写入文件
    }

    // 异步保存（fork子进程）
    void bg_save() {
        if (bg_save_in_progress_) return;  // 防止重复BGSAVE

        child_pid_ = fork();
        if (child_pid_ == 0) {
            // 子进程
            save();
            _exit(0);
        } else {
            // 父进程
            bg_save_in_progress_ = true;
        }
    }
};
```

#### 验收标准

- RDB文件能正确生成
- 文件格式与Redis兼容
- 数据在保存后保持完整

### 4.4.4 RDB文件加载

#### 为什么需要RDB加载？

服务器启动时能够恢复内存数据：

| 场景 | 说明 |
|-----|------|
| 服务器重启 | 执行kill -9后重启，加载RDB恢复数据 |
| 主从切换 | 从节点重启后，加载RDB与主节点数据一致 |
| 数据迁移 | 将RDB文件拷贝到新服务器，直接加载使用 |
| 紧急恢复 | 发现数据异常时，加载历史快照回溯数据 |

#### 加载流程

```
1. 检查RDB文件是否存在
2. 打开文件，验证MAGIC
3. 读取版本号
4. 读取辅助字段
5. 遍历数据库：
   - 读取SELECTDB和数据库编号
   - 读取键值对并恢复到内存
6. 验证CRC64
```

#### 实现要点

```cpp
class RDBLoader {
public:
    bool load(const std::string& filename) {
        std::ifstream file(filename, std::ios::binary);
        if (!file) return false;

        // 验证MAGIC
        char magic[10];
        file.read(magic, 9);
        if (std::string(magic, 9) != "REDIS0009") return false;

        // 读取版本号
        int version = read_int(file);

        // 读取辅助字段
        read_aux_fields(file);

        // 读取数据库
        while (true) {
            int type = file.get();
            if (type == EOF) break;
            if (type == 0xFF) break;  // EOF

            if (type == 0xFE) {  // SELECTDB
                int db_id = read_length(file);
                switch_database(db_id);
                continue;
            }

            // 读取键值对
            auto key = read_string(file);
            auto value = read_value(file, type);
            storage_->set(key, value);
        }

        // 验证CRC64
        uint64_t expected_crc = read_crc64(file);
        uint64_t actual_crc = calculate_crc64(file);
        return expected_crc == actual_crc;
    }
};
```

#### 验收标准

- 服务器启动时能正确加载RDB文件
- 加载后数据完整
- 损坏的RDB文件能正确报错

## 4.5 章节4：AOF持久化

### 4.5.1 为什么需要AOF？

AOF（Append Only File）追加日志持久化将每个写命令记录到文件：

| 特性 | 说明 |
|-----|------|
| **数据安全性更高** | 可以配置每个写操作都刷盘 |
| **实时持久化** | 相比RDB的定时快照，AOF可以更实时地保存数据 |
| **命令可读** | AOF文件是文本格式，可以直接查看和编辑 |
| **日志价值** | 可以用于分析操作历史、审计等 |

#### AOF vs RDB

| 特性 | RDB | AOF |
|-----|-----|-----|
| 文件大小 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢（需要重放命令） |
| 数据安全性 | 可能丢失快照后的数据 | 可配置（always/everysec/no） |
| 写入方式 | 定时快照 | 追加写入 |

---

### 4.5.2 刷盘策略

| 策略 | 说明 | 安全性 | 性能 |
|-----|------|-------|-----|
| always | 每个命令都刷盘 | 最高 | 最低 |
| everysec | 每秒刷盘一次 | 中等 | 较高 |
| no | 由操作系统决定 | 最低 | 最高 |

#### 为什么everysec是折中方案？

```
always的问题：
┌─────────────────────────────────────────────────────────────┐
│  每次写命令都要fsync                                        │
│  → 大量小I/O操作                                           │
│  → 磁盘成为瓶颈                                            │
│  → QPS大幅下降                                            │
└─────────────────────────────────────────────────────────────┘

everysec的优化：
┌─────────────────────────────────────────────────────────────┐
│  每秒一次批量刷盘                                          │
│  → 合并1秒内的多次写入                                     │
│  → 减少I/O次数                                            │
│  → 最多丢失1秒数据                                         │
└─────────────────────────────────────────────────────────────┘
```

---

### 4.5.3 AOF文件写入

#### 实现要点

```cpp
class AOFPersistence {
private:
    std::string filename_;
    AppendFSyncMode mode_;  // always/everysec/no
    std::ofstream file_;
    std::mutex mutex_;
    std::atomic<bool> flush_in_progress_;

public:
    void append_command(const std::vector<std::string>& args) {
        // 将命令序列化为RESP协议格式
        std::string cmd = encode_as_resp(args);

        std::lock_guard<std::mutex> lock(mutex_);
        file_.write(cmd.data(), cmd.size());

        if (mode_ == ALWAYS) {
            file_.flush();  // 立即刷盘
        }
    }
};
```

- 三种刷盘策略都正常工作

### 4.5.4 AOF重写

#### 问题分析

```
AOF文件膨胀：
┌─────────────────────────────────────────────────────────────┐
│  原始AOF文件（1年后）：                                      │
│  SET key1 v1    ← 1年前                                    │
│  SET key1 v2    ← 11个月前                                 │
│  SET key1 v3    ← 10个月前                                 │
│  ...                                                        │
│  SET key1 v100   ← 现在                                    │
│                                                              │
│  问题：                                                    │
│  - 只有最后一个SET key1 v100是有意义的                      │
│  - 中间的99个SET都是冗余的                                 │
│  - 文件大小 = 100 × sizeof("SET key1 vN")                 │
│  - 实际只需要 = 1 × sizeof("SET key1 v100")                │
└─────────────────────────────────────────────────────────────┘
```

#### 重写原理

```
原始AOF文件：
SET key1 v1    # 被后续命令覆盖
SET key1 v2    # 被后续命令覆盖
SET key1 v3    # 最新值，保留
DEL key2       # 被DEL后无意义
SET key2 new_value  # 有意义，保留

重写后AOF：
SET key1 v3
SET key2 new_value
```

#### 为什么需要后台重写？

```
前台重写的问题：
┌─────────────────────────────────────────────────────────────┐
│  重写过程中：                                                │
│  → 无法处理请求                                            │
│  → 服务中断                                                 │
└─────────────────────────────────────────────────────────────┘

后台重写（BGREWRITEAOF）：
┌─────────────────────────────────────────────────────────────┐
│  fork子进程执行重写                                         │
│  主进程继续处理请求                                         │
│  重写期间的新命令写入旧的AOF文件                            │
│  重写完成后，用新文件替换旧文件                             │
└─────────────────────────────────────────────────────────────┘
```

#### 验收标准

- AOF重写能有效减小文件大小
- 重写过程中不影响服务
- 重写后的AOF能正确恢复数据

### 4.5.5 混合持久化

#### 为什么需要混合持久化？

混合持久化结合RDB和AOF的优点：

| 特性 | 说明 |
|-----|------|
| **最优恢复** | 同时存在时优先加载AOF（数据更完整） |
| **快速恢复** | 先加载RDB作为基础，再重放AOF增量 |
| **文件优化** | 重写AOF时使用RDB格式作为前缀 |

#### 加载优先级

```
1. 如果AOF存在且非空 → 优先加载AOF
2. 否则如果RDB存在 → 加载RDB
3. 如果都不存在 → 从空状态启动
```

#### 实现要点

```cpp
void PersistenceManager::load() {
    // 优先加载AOF
    if (file_exists("appendonly.aof")) {
        if (load_aof("appendonly.aof")) {
            return;
        }
    }

    // 否则加载RDB
    if (file_exists("dump.rdb")) {
        load_rdb("dump.rdb");
    }
}
```

- 混合持久化功能正常
- AOF优先加载
- 数据完整性最高

### 4.6.5 CPU优化

#### 优化方向

| 问题 | 解决方案 |
|-----|---------|
| 上下文切换 | CPU绑定（affinity） |
| 缓存未命中 | 数据对齐、预取 |
| 分支预测失败 | likely/unlikely宏 |

#### SubReactor线程绑定

```cpp
void SubReactor::run() {
    // 绑定到特定CPU核心
    cpu_set_t cpu_set;
    CPU_ZERO(&cpu_set);
    CPU_SET(thread_index_ % std::thread::hardware_concurrency(), &cpu_set);
    pthread_setaffinity_np(pthread_self(), sizeof(cpu_set), &cpu_set);

    // 事件循环
    while (running_) {
        event_loop_.loop();
    }
}
```

#### 验收标准

- 上下文切换减少
- 单请求处理延迟降低
