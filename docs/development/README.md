# 第一章：迭代路线与开发理念

## 1.1 故事线：为什么要开发这个项目？

### 起点：遇到的问题

假设你是一个后端工程师，你的系统需要缓存热点数据来提升性能。你听说 Redis 很好用，于是你想：

```
系统 A (Go) ──────► Redis (6379) ──────► 存储
系统 B (Python) ───► Redis (6379) ──────► 存储
系统 C (Java) ────► Redis (6379) ──────► 存储
```

**但是问题来了：**

| 问题 | 说明 |
|-----|------|
| **Redis 太重了** | 你只是想要一个简单的键值缓存，不需要 Redis 的全部功能 |
| **运维成本** | 需要单独部署和维护 Redis 进程 |
| **学习目的** | 你想深入理解高并发服务器是怎么工作的 |

于是你决定：**自己写一个简单的缓存服务器**。

## 1.3 迭代决策的因果链条

### 为什么先做骨架版本？

```
问题：从来没有实现过一个完整的服务器程序
     │
     ▼
分析：如果一开始就做多线程、内存池、分布式
     - 代码复杂度爆炸
     - 问题难以定位
     - 无法理解核心组件的协作关系
     │
     ▼
决策：先做最小可运行版本（骨架版本）
     - 单线程、简单架构
     - 只实现核心功能
     - 确保每个组件都能正常工作
     │
     ▼
结果：建立对整个系统的直觉，理解组件如何协作
```

### 为什么骨架版本之后要做基础版本？

```
骨架版本的问题：
┌─────────────────────────────────────────────────────────────┐
│  单Reactor只能利用一个CPU核心                                 │
│  → 处理能力有限（10万连接/秒是极限）                          │
│                                                              │
│  malloc/free在高并发下性能差                                   │
│  → 每次分配/释放都要系统调用，锁竞争严重                      │
│                                                              │
│  全局锁哈希表                        │
│  → 多线程并发读写时锁竞争严重                                │
│                                                              │
│  没有过期淘汰机制                                            │
│  → 内存会无限增长                                            │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
分析：这些都是"跑得快"的问题
     - 骨架版本解决了"能跑"
     - 基础版本要解决"跑得快、跑得稳"
     │
     ▼
决策：基础版本专注于性能基础设施
     - 多线程网络模型（MainSubReactor）
     - 内存池
     - 高性能哈希表
     - 缓存淘汰策略
```

### 为什么基础版本之后要做增强版本？

```
基础版本的问题：
┌─────────────────────────────────────────────────────────────┐
│  只支持String类型                                            │
│  → 无法满足复杂业务场景（用户信息、排行榜、消息队列）          │
│                                                              │
│  没有持久化                                                  │
│  → 进程重启数据全部丢失                                     │
│                                                              │
│  只有一个淘汰算法（LRU）                                     │
│  → 不同业务场景需要不同策略                                  │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
分析：这些都是"功能完备性"的问题
     - 基础版本解决了"基础设施"
     - 增强版本要解决"功能丰富性"
     │
     ▼
决策：增强版本添加
     - 更多数据结构（Hash/List/Set/ZSet）
     - 持久化机制（RDB/AOF）
     - 多种淘汰算法（LFU/FIFO/Random）
```

### 为什么增强版本之后要做分布式版本？

```
增强版本的问题：
┌─────────────────────────────────────────────────────────────┐
│  单机存储容量受内存限制                                       │
│  → 无法存储海量数据                                          │
│                                                              │
│  单点故障                                                    │
│  → 进程崩溃服务就中断                                        │
│                                                              │
│  无法水平扩展                                                │
│  → 只能垂直升级硬件                                          │
└─────────────────────────────────────────────────────────────┘
     │
     ▼
分析：这些都是"分布式"的问题
     - 增强版本是"单机版终极形态"
     - 分布式版本要解决"扩展性和可用性"
     │
     ▼
决策：分布式版本实现
     - 数据分片（16384哈希槽）
     - 主从复制
     - 自动故障转移
     - Gossip协议
```

## 1.4 核心设计理念

### 1.4.1 事件驱动 vs 多线程

**为什么选择事件驱动（EventLoop + epoll）而不是多线程？**

```
多线程方案的问题：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 线程1   │ │ 线程2   │ │ 线程3   │   ← 每个线程都要占用栈内存
│ 处理A   │ │ 处理B   │ │ 处理C   │
│ 等待I/O │ │ 等待I/O │ │ 等待I/O │   ← 大部分时间在等待，浪费CPU
└─────────┘ └─────────┘ └─────────┘

事件驱动方案的优势：
┌─────────────────────────────────────┐
│           EventLoop (单线程)        │
│                                     │
│   epoll_wait() 阻塞等待 ──────► 有事件才处理
│                                     │
│   处理 A ──► 处理 B ──► 处理 C       │  ← CPU 始终在工作
└─────────────────────────────────────┘

结论：对于 IO 密集型任务（网络服务器大部分时间在等数据），事件驱动更高效。
```

### 1.4.2 为什么需要模块化分层？

```
不分层的问题：
┌─────────────────────────────────────────┐
│  main() {                             │
│      accept();                         │
│      parse_command();                  │
│      execute_storage();               │
│      send_response();                 │
│  }                                    │
│                                         │
│  问题：所有逻辑混在一起，改一个功能要   │
│  理解全部代码，难以维护和扩展          │
└─────────────────────────────────────────┘

分层的优势：
┌─────────────────────────────────────────┐
│  应用层：命令处理（用户能看到的）       │
├─────────────────────────────────────────┤
│  协议层：数据解析（如何理解数据）       │
├─────────────────────────────────────────┤
│  存储层：数据存储（如何保存数据）       │
├─────────────────────────────────────────┤
│  网络层：通信传输（如何收发数据）       │
└─────────────────────────────────────────┘

优势：
- 每层只关注自己的事
- 可以独立测试、替换
- 易于理解和维护
```

### 1.4.3 为什么要用 RAII 和智能指针？

```
手动管理内存的问题：
void handle_request() {
    Connection* conn = new Connection(fd);
    // ... 处理逻辑 ...
    if (error) {
        delete conn;  // 忘记 delete？内存泄漏！
        return;
    }
    // ... 更多代码 ...
    delete conn;  // 重复 delete？崩溃！
}

智能指针方案（安全）：
void handle_request() {
    auto conn = std::make_unique<Connection>(fd);
    // ... 处理逻辑 ...
    if (error) {
        return;  // conn 自动销毁
    }
    // ... 更多代码 ...
    return;  // conn 自动销毁
}
```

## 1.6 文档使用建议

