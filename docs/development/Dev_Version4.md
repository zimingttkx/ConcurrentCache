# ConcurrentCache 分布式版本（Version4）

## 版本概述

**目标**：实现分布式集群功能，支持水平扩展和高可用。

分布式版本是项目的终极目标，实现Redis Cluster风格的分片集群，支持大规模数据存储和高可用。

**预计开发周期**：12-16周（可选）

**核心功能**：
- 哈希槽分片（16384个槽）
- 主从复制（全量复制+增量复制）
- 自动故障转移
- Gossip协议通信
- 集群节点管理

---

## 组件作用说明

### 为什么要开发分布式版本？

单机缓存有明显的局限性：

| 局限性 | 单机版 | 分布式版 |
|-------|-------|---------|
| **存储容量** | 受单机内存限制 | 横向扩展，容量无上限 |
| **可用性** | 单点故障 | 主从复制，自动故障转移 |
| **性能** | 受单机CPU/网络限制 | 多节点并行处理 |
| **扩展性** | 只能垂直扩展 | 支持水平扩展 |

**分布式带来的核心价值**：

1. **水平扩展**：通过增加节点来扩展容量和性能
2. **高可用**：主从复制保证数据不丢失，自动故障转移保证服务连续性
3. **数据分片**：16384个槽分散到多个节点，充分利用集群资源
4. **负载均衡**：请求自动路由到合适的节点

## 章节1：集群基础（第1-2周）

### 任务1.1：集群配置

**目标**：实现集群配置和节点初始化

**为什么需要这个组件？**

集群配置是分布式集群的基础，定义了集群的拓扑结构和节点关系：

1. **节点标识**：每个节点有唯一ID，IP+端口可变化，节点ID不变
2. **角色定义**：主节点（master）负责存储槽，从节点（replica）负责复制主节点
3. **拓扑管理**：维护集群中所有节点的信息

**节点角色的作用**：
| 角色 | 职责 | 数量 |
|-----|------|-----|
| Master | 负责存储槽，处理请求 | 至少1个 |
| Replica | 复制主节点数据，提供读服务 | 可选 |

**为什么需要节点ID？**
```
节点IP可能变化（如容器重启后IP改变），但节点ID保持不变
这样集群可以识别"这是同一个节点"，不需要重新加入集群
```

**实现要点**：
1. 集群节点配置（节点ID、IP地址、端口、角色）
2. 集群节点启动参数
3. 节点信息持久化
4. 集群拓扑结构管理

**配置项**：
```
cluster-enabled = yes
cluster-node-timeout = 15000        # 节点超时时间（ms）
cluster-replica-validity-factor = 10 # 从节点故障转移条件
cluster-migration-barrier = 1        # 迁移时主节点最少从节点数
```

**类设计**：
```cpp
class ClusterConfig {
public:
    std::string node_id_;
    std::string ip_;
    int port_;
    NodeRole role_;  // MASTER/REPLICA
    std::set<uint16_t> slots_;  // 负责的槽
};

class Cluster {
public:
    void add_node(const ClusterConfig& config);
    void remove_node(const std::string& node_id);
    ClusterConfig* get_node(const std::string& node_id);
    const std::map<std::string, ClusterConfig>& get_nodes() const;
};
```

**验收标准**：
- 集群配置正确读取
- 节点信息正确管理

### 任务1.2：槽映射管理

**目标**：实现哈希槽到节点的映射

**为什么需要这个组件？**

槽映射是集群数据分片的核心，决定了数据如何分布在集群中：

1. **数据分片**：16384个槽分散在不同节点，实现数据分布存储
2. **键定位**：通过CRC16算法快速计算键属于哪个槽
3. **负载均衡**：槽均匀分布保证各节点负载大致相同
4. **扩缩容基础**：槽迁移是扩缩容的基础

**为什么选择16384个槽？**
```
- 槽数太少：扩缩容时迁移的键太多，影响服务
- 槽数太多：管理开销增大，每个节点需要维护更多槽信息
- 16384是2^14，适中：每个主节点平均负责约1700个槽

选择依据：
- 节点数 × 每节点槽数 ≈ 16384
- 例如：3个主节点 × 5461槽/节点 ≈ 16384
```

**CRC16算法原理**：
```
键 → CRC16(key) → 取模16384 → 槽号

CRC16（Redis标准多项式）：
- 输入：任意长度的key字符串
- 输出：0-65535的值
- 取模16384得到槽号

示例：
"user:1000" → CRC16 → 12345 → 12345 % 16384 → 槽12345
```

**槽分配示例**：
```
集群3个主节点，槽分配如下：
Node1: 槽0-5460     (5461个槽)
Node2: 槽5461-10922 (5462个槽)
Node3: 槽10923-16383 (5461个槽)

键"user:1000"计算得槽12345 → Node2处理
```

**实现要点**：
1. 16384个槽的映射管理
2. 键到槽的映射：CRC16(key) % 16384
3. 槽到节点的映射
4. 槽分配信息持久化

**槽映射算法**：
```
slot = CRC16(key) % 16384

CRC16算法：
- 使用Redis标准的CRC16多项式
- 输入：key字符串
- 输出：0-16383的槽号
```

**类设计**：
```cpp
class SlotMap {
public:
    static uint16_t key_to_slot(const std::string& key);
    void set_slot_owner(uint16_t slot, const std::string& node_id);
    std::string get_slot_owner(uint16_t slot);
    void add_slot(uint16_t slot);
    void remove_slot(uint16_t slot);
    const std::set<uint16_t>& get_node_slots(const std::string& node_id);

private:
    std::array<std::string, 16384> slot_to_node_;  // 槽到节点的映射
    std::map<std::string, std::set<uint16_t>> node_to_slots_;  // 节点到槽的反向映射
};
```

**验收标准**：
- 键能正确映射到槽
- 槽分配信息正确管理

**为什么需要这个组件？**

集群节点之间需要相互通信来同步状态，TCP连接是通信的基础：

1. **Gossip通信**：节点间交换状态信息

### 任务2.2：Gossip协议

**目标**：实现Gossip协议用于节点状态同步

**为什么需要这个组件？**

Gossip协议是分布式系统中常用的状态同步协议，类比"八卦传播"：

1. **去中心化**：无需中心节点，每个节点都可以传播信息
2. **最终一致**：信息逐渐传播到所有节点
3. **容错性**：节点宕机不影响其他节点通信
4. **可扩展性**：节点越多，传播越均匀

**Gossip传播示意图**：
```
初始：Node1知道新节点信息
       Node1
         ↓ Gossip
       Node2、Node3得知

       Node2、Node3
         ↓ Gossip
       Node4、Node5、Node6得知

       ... 信息逐渐传播到所有节点
```

**Gossip消息类型说明**：
| 消息类型 | 作用 | 触发时机 |
|---------|------|---------|
| MEET | 新节点加入集群 | CLUSTER MEET命令 |
| PING | 主动探测节点存活 | 定期（每秒随机选节点） |
| PONG | 响应探测 | 收到PING后 |
| FAIL | 节点故障通知 | 主观下线时 |
| FAILOVER | 故障转移通知 | 从节点升级时 |
| UPDATE | 槽分配更新 | 槽迁移完成时 |

**为什么Gossip比中心化更好？**
```
中心化方案（不好）：
- 中心节点宕机 → 整个集群通信瘫痪
- 中心节点压力大 → 可能成为瓶颈

Gossip方案（好）：
- 无中心节点 → 任何节点宕机都不影响
- 负载分散 → 每节点只与少数节点通信
```

**实现要点**：
1. Gossip消息格式定义
2. 节点信息交换（meet、ping、pong）
3. 定期广播节点状态
4. 消息压缩

**Gossip消息类型**：
```
MEET        # 新节点加入集群
PING        # 主动探测节点
PONG        # 响应探测
FAIL        # 节点故障
FAILOVER    # 故障转移
UPDATE      # 槽分配更新
```

**Gossip消息格式**：
```protobuf
message GossipMessage {
    MessageType type = 1;
    string sender_id = 2;
    uint64_t cluster_epoch = 3;
    repeated NodeInfo nodes = 4;
}

message NodeInfo {
    string node_id = 1;
    string ip = 2;
    int port = 3;
    NodeRole role = 4;
    uint16_t slot_start = 5;
    uint16_t slot_end = 6;
    uint64_t last_pong_time = 7;
    uint32_t failure_count = 8;
}
```

**验收标准**：
- Gossip消息正确发送和接收
- 节点状态能正确同步

### 任务2.3：节点发现

**目标**：实现新节点发现和加入集群

**为什么需要这个组件？**

节点发现让新节点能够自动融入集群，是集群动态扩展的基础：

1. **引导机制**：新节点只需要知道集群中任意一个节点
2. **自动传播**：通过Gossip协议，信息自动传播到所有节点
3. **槽分配同步**：新节点加入后能获取当前的槽分配信息

**节点加入流程**：
```
新节点启动时：
1. 配置文件或参数指定已知节点IP:Port
2. 新节点发送CLUSTER MEET到已知节点
3. 已知节点将新节点加入自己的节点列表
4. 已知节点通过Gossip广播新节点信息
5. 所有节点陆续得知新节点存在
6. 新节点通过Gossip学习到完整的集群拓扑
```

**CLUSTER MEET命令的作用**：
```
# 新节点执行：
CLUSTER MEET 192.168.1.10 6379

# 相当于告诉集群：
"请把我介绍给集群，我是新成员"
```

**为什么只需要知道一个节点？**
```
不需要知道所有节点，因为：
- 通过Gossip协议，一个节点知道的信息会传播给其他节点
- 第一个节点会告诉第二个，第二个会告诉第三个...
- 最终所有节点都知道新节点的存在
```

**实现要点**：
1. CLUSTER MEET命令实现
2. 节点引导（指定一个已知节点）
3. 节点信息传播到集群所有节点
4. 槽分配信息同步

**节点发现流程**：
```
1. 新节点发送CLUSTER MEET到已知节点
2. 已知节点将新节点信息通过Gossip广播
3. 所有节点陆续得知新节点存在
4. 新节点开始接收请求和Gossip消息
```

**验收标准**：
- 新节点能正确加入集群
- 所有节点都能发现新节点

## 章节3：数据分片（第5-6周）

### 任务3.1：请求路由

**目标**：实现请求到正确节点的路由

**为什么需要这个组件？**

客户端请求到达任意节点，但数据可能不在该节点上，需要正确路由：

1. **本地槽处理**：本节点负责的槽，直接处理
2. **MOVED重定向**：槽已迁移，客户端更新映射
3. **ASK重定向**：槽迁移中，临时转向目标节点
4. **客户端缓存**：客户端缓存槽映射减少重定向

**MOVED vs ASK重定向**：
| 类型 | 原因 | 客户端行为 | 后续请求 |
|-----|------|-----------|---------|
| MOVED | 槽永久迁移到新节点 | 更新本地槽映射 | 直接请求新节点 |
| ASK | 槽正在迁移中（迁移还没完成） | 临时转向新节点 | 继续请求源节点 |

**为什么需要两种重定向？**
```
MOVED：槽已经迁移完成
- 告诉客户端槽的新位置
- 客户端更新缓存，下次直接访问新节点

ASK：槽正在迁移中（迁移还没完成）
- 源节点还有数据，目标节点也有部分数据
- 如果客户端收到ASK后直接访问目标节点
- 源节点数据会丢失
- 所以ASK只做临时转向，不更新客户端映射
```

**请求路由流程**：
```
客户端发送 GET user:1000
    ↓
Node1收到请求
    ↓
计算槽号：CRC16("user:1000") % 16384 = 12345
    ↓
Node1查槽12345归属 → Node2
    ↓
12345在Node1本地？ → 否
    ↓
返回 MOVED 12345 192.168.1.20:6379
    ↓
客户端收到MOVED，更新映射
    ↓
下次直接请求Node2
```

**实现要点**：
1. 本地处理本节点负责的槽
2. 非本地槽返回MOVED重定向
3. ASK重定向（槽迁移中）
4. 客户端缓存槽映射（可选）

**响应格式**：
```
MOVED <slot> <ip>:<port>    # 槽已迁移，客户端应更新映射
ASK <slot> <ip>:<port>      # 槽迁移中，客户端应临时转向
```

**类设计**：
```cpp
class RequestRouter {
public:
    Response route(const std::string& key, const Command& cmd);
    bool is_local_slot(uint16_t slot);
    std::string get_slot_owner(uint16_t slot);

private:
    SlotMap* slot_map_;
    Cluster* cluster_;
    Connection* find_connection(const std::string& node_id);
};
```

**验收标准**：
- 本地槽请求正确处理
- 非本地槽返回正确的MOVED/ASK响应

### 任务3.2：槽迁移

**目标**：实现槽在节点之间的迁移

**为什么需要这个组件？**

槽迁移是集群扩缩容的核心，保证服务在扩缩容期间不中断：

1. **动态扩缩容**：添加/移除节点时重新分配槽
2. **在线迁移**：迁移过程中服务不中断
3. **数据完整性**：确保迁移过程中数据不丢失
4. **状态透明**：客户端无感知

**槽迁移流程**：
```
场景：Node1的槽12345需要迁移到Node2

阶段1：开始迁移
- Node1执行 CLUSTER SETSLOT 12345 MIGRATING Node2
- Node1标记槽12345为迁移中状态
- Node2执行 CLUSTER SETSLOT 12345 IMPORTING Node1
- Node2标记槽12345为迁入状态

阶段2：数据迁移
- Node1遍历槽12345的所有键
- 将每个键值对发送到Node2
- Node2接收并存储这些数据

阶段3：完成迁移
- Node1确认所有数据已迁移
- 执行 CLUSTER SETSLOT 12345 Node2
- 所有节点更新槽映射，Node1不再负责槽12345
```

**迁移中的请求处理**：
```
客户端请求槽12345的数据：

情况1：客户端请求到Node1（旧节点）
- Node1发现槽12345正在迁移
- 返回 ASK 12345 Node2:6379
- 客户端临时转向Node2

情况2：客户端请求到Node2（新节点）
- Node2发现槽12345是迁入状态
- 处理请求
- 如果数据不存在，向Node1获取

情况3：迁移完成后
- 槽12345已完全属于Node2
- 返回 MOVED 12345 Node2:6379
```

**为什么迁移过程不会丢失数据？**
```
关键：ASK只做临时转向，不更新客户端映射

客户端访问Node1（迁移中）：
1. Node1检查数据是否已迁移 → 否
2. 返回ASK，指向Node2
3. 客户端转向Node2获取数据
4. Node2没有数据时，向Node1请求

这样确保：
- 迁移过程中的请求都能正确处理
- 客户端映射不会在迁移中途更新
- 数据完整性得到保证
```

**实现要点**：
1. 槽迁移状态管理（迁移中/已完成）
2. 迁移过程中的请求处理（ASK重定向）
3. 批量迁移键值对
4. 迁移完成确认

**槽迁移流程**：
```
1. 源节点：标记槽为迁移中状态
2. 源节点：设置迁移目标节点信息
3. 目标节点：开始接收该槽的键
4. 源节点：发送键值对到目标节点
5. 源节点：发送确认完成消息
6. 所有节点：更新槽映射
```

**迁移状态**：
```cpp
struct SlotMigration {
    uint16_t slot;
    std::string source_node;
    std::string target_node;
    MigrationState state;  // START/IN_PROGRESS/COMPLETE
};
```

**验收标准**：
- 槽迁移能正确完成
- 迁移过程中数据不丢失
- 迁移过程中请求能正确处理

### 任务3.3：动态扩缩容

**目标**：支持集群动态扩缩容

**为什么需要这个组件？**

分布式系统需要能够动态调整容量，扩缩容是核心能力：

1. **水平扩展**：通过添加节点增加容量
2. **缩容支持**：节点移除时自动迁移数据
3. **负载均衡**：槽均匀重新分配
4. **业务无感知**：扩缩容过程服务不中断

**扩容场景示例**：
```
3节点集群扩容到4节点：

扩容前（3节点）：
Node1: 槽0-5460
Node2: 槽5461-10922
Node3: 槽10923-16383

扩容步骤：
1. 新节点Node4加入集群（此时没有槽）
2. 从各主节点迁移一部分槽到Node4
3. 最终每个节点约4096个槽

扩容后（4节点）：
Node1: 槽0-4095
Node2: 槽4096-8191
Node3: 槽8192-12287
Node4: 槽12288-16383
```

**缩容场景示例**：
```
4节点集群缩容到3节点：

缩容步骤：
1. 将Node4的槽12388-16383迁移到其他节点
2. Node4所有槽迁出后，从集群移除
3. 其他节点槽数增加

缩容后（3节点）：
Node1: 槽0-5460
Node2: 槽5461-10922
Node3: 槽10923-16383
```

**扩缩容对业务的影响**：
```
扩缩容前：
3节点集群，QPS 10万

扩缩容中：
- 槽迁移是后台进行的
- 正常请求通过MOVED/ASK重定向处理
- 对业务有一定影响（约10-20%性能波动）

扩缩容后：
4节点集群，QPS 13万（理论线性扩展）
```

**实现要点**：
1. 添加新节点到集群
2. 重新分配槽
3. 移除节点
4. 槽迁移协调

**验收标准**：
- 新节点能加入并分配槽
- 节点移除后槽能重新分配
- 扩缩容过程不影响服务

## 章节4：主从复制（第7-9周）

### 任务4.1：复制配置

**目标**：实现主从节点配置和初始化

**为什么需要这个组件？**

主从复制是高可用的基础，保证数据不丢失和服务持续可用：

1. **数据冗余**：从节点保存主节点的数据副本
2. **读写分离**：从节点可提供读服务，分担主节点压力
3. **故障恢复**：主节点故障时从节点可升级
4. **异地灾备**：从节点可部署在不同机房

**主从复制架构**：
```
                    客户端写入
                        ↓
┌─────────────────── 主节点(Master) ───────────────────┐
│                      数据存储                          │
│                         ↓ 复制                         │
│  ┌─────────────────┐  ┌─────────────────┐            │
│  │  从节点(Replica) │  │  从节点(Replica) │            │
│  │   数据副本       │  │   数据副本       │            │
│  └─────────────────┘  └─────────────────┘            │
└───────────────────────────────────────────────────────┘

写入流程：客户端 → 主节点 → 复制 → 从节点
读取流程（可负载均衡）：客户端 → 从节点
```

**replicaof配置的作用**：
```
# 从节点执行：
replicaof 192.168.1.10 6379

# 效果：
- 从节点开始复制指定主节点
- 建立主从复制链路
- 接收主节点的增量命令
```

**实现要点**：
1. 从节点配置（主节点信息）
2. 主节点信息持久化
3. 复制链路建立

**配置项**：
```
replicaof <master_ip> <master_port>  # 配置主节点
```

**类设计**：
```cpp
class ReplicationConfig {
public:
    std::string master_ip_;
    int master_port_;
    std::string master_node_id_;
    ReplicationState state_;  // CONNECTING/CONNECTED/TRANSFERING
};
```

**验收标准**：
- 从节点能正确配置主节点

### 任务4.2：全量复制

**目标**：实现主从全量复制

**为什么需要这个组件？**

从节点刚加入或复制链路断开时，需要全量复制来同步数据：

1. **数据初始化**：从节点第一次同步，需要完整数据
2. **断连恢复**：复制链路断开后的重新同步
3. **保证一致性**：确保从节点数据与主节点完全一致
4. **增量基础**：全量复制后的增量复制

**全量复制流程**：
```
从节点视角：
1. 发送 PSYNC ? -1
   （？表示不知道主节点runid，-1表示全量复制）

主节点视角：
2. 返回 +FULLRESYNC <runid> <offset>
   （告诉从节点全量复制，并用runid和offset标识）

3. 执行BGSAVE生成RDB文件
   （fork子进程，父进程继续处理请求）

4. RDB文件生成后，发送给从节点
   （可能很大，需要分片传输）

5. 发送RDB文件期间的增量命令
   （主节点将这些命令存入复制缓冲区）

从节点视角：
6. 接收RDB文件并加载到内存
7. 接收增量命令并重放
8. 复制开始，正常接收增量命令
```

**PSYNC命令解析**：
```
PSYNC <runid> <offset>

runid：
- ? 表示第一次同步，需要全量复制
- 主节点runid 表示增量复制（部分同步）

offset：
- -1 表示全量复制
- 正整数表示从该偏移量开始增量复制
```

**实现要点**：
1. 主从节点握手（PING/PONG）
2. 主节点生成RDB文件
3. RDB文件发送给从节点
4. 从节点加载RDB文件
5. 复制偏移量同步

**全量复制流程**：
```
1. 从节点发送PSYNC ? -1
2. 主节点返回+FULLRESYNC <runid> <offset>
3. 主节点BGSAVE生成RDB
4. 主节点发送RDB文件到从节点
5. 从节点加载RDB文件
6. 主节点发送缓冲区中的增量命令
7. 复制开始
```

**验收标准**：
- 全量复制正常工作
- 数据完整同步

### 任务4.3：增量复制

**目标**：实现主从增量复制

**为什么需要这个组件？**

全量复制代价很高（需要生成和传输RDB文件），增量复制让正常情况下的同步更高效：

1. **低开销**：只传输增量命令，无需传输整个数据集
2. **高效率**：网络传输量小，同步延迟低
3. **高吞吐**：主节点可以支持更多从节点
4. **积压缓冲区**：记录近期命令，支持断线重连

**增量复制工作原理**：
```
主节点：
1. 每个写命令执行时，同步写入两个地方：
   - 客户端返回（正常处理）
   - 复制积压缓冲区（环形缓冲区）

2. 定期PING从节点，携带当前复制偏移量

从节点：
3. 定期发送REPLCONF ACK <offset>
   （告诉主节点，我已经同步到哪个位置了）

4. 如果从节点偏移量落后，主节点补发缺失的命令
```

**复制积压缓冲区**：
```
环形缓冲区结构（默认1MB）：

┌────────────────────────────────────────────────────────┐
│ cmd1 │ cmd2 │ cmd3 │ cmd4 │ cmd5 │ ... │ cmdN │     │
└────────────────────────────────────────────────────────┘
 ↑写指针                                      ↑读指针

- 新命令写入：写指针向前移动
- 从节点确认：读指针向前移动
- 缓冲区满：覆盖最旧的数据

如果从节点断开重连：
- 从节点发送 REPLCONF ACK <offset>
- 主节点检查从节点偏移量
- 如果在积压缓冲区范围内 → 只发送缺失的命令
- 如果不在范围内 → 需要全量复制
```

**实现要点**：
1. 主节点记录写命令到复制积压缓冲区
2. 主节点发送PING心跳（携带复制偏移量）
3. 从节点发送REPLCONF ACK <offset>
4. 主节点发送缺失的命令

**复制积压缓冲区**：
```
环形缓冲区：
- 大小：默认1MB
- 记录每个写命令和偏移量
- 定期清理过期数据
```

**验收标准**：
- 增量复制正常工作
- 主从数据最终一致

### 任务4.4：只读从节点

**目标**：实现从节点只读模式

**为什么需要这个组件？**

从节点默认只读，既保证了数据一致性，又提高了系统可用性：

1. **数据一致性**：防止从节点被意外写入导致数据不一致
2. **读写分离**：读请求分流到从节点，减轻主节点压力
3. **故障转移**：从节点可升级为主节点，不应包含写操作
4. **安全防护**：防止应用程序误写入从节点

**只读模式的行为**：
```
客户端尝试在从节点执行写命令：
127.0.0.1:6380> SET key value
(error) READONLY You can't write against a read only replica.

只读从节点正确响应读命令：
127.0.0.1:6380> GET key
"value"
127.0.0.1:6380> EXISTS key
(integer) 1
```

**为什么从节点需要只读？**
```
1. 数据流向单一：
   主节点 ← 复制 ← 从节点
   从节点写入 → 数据不一致 → 集群状态混乱

2. 故障转移场景：
   从节点升级为主节点后，原主节点重新加入成为从节点
   如果从节点可写 → 可能写入丢失 → 数据丢失

3. 读写分离场景：
   大量读请求分散到多个从节点
   如果从节点可写 → 读负载分散效果差
```

**实现要点**：
1. 从节点默认只读
2. 拒绝写命令
3. 复制命令正常执行

**验收标准**：
- 从节点能处理读命令
- 从节点拒绝写命令

## 章节5：故障转移（第10-12周）

### 任务5.1：故障检测

**目标**：实现节点故障检测

**为什么需要这个组件？**

故障检测是故障转移的前提，快速准确地检测故障才能及时恢复：

1. **心跳机制**：通过PING/PONG检测节点存活
2. **主观下线**：单个节点认为另一个节点故障
3. **客观下线**：集群认为某个节点真正故障
4. **故障传播**：故障信息广播到所有节点

**为什么需要主观+客观两层检测？**
```
主观下线（单节点判断）：
- 节点A在15秒内没收到节点B的PONG
- 节点A认为节点B故障
- 但这可能只是网络抖动或节点B负载高

客观下线（集群判断）：
- 超过半数的节点都认为节点B故障
- 才认定节点B真正下线
- 避免误判导致的频繁故障转移
```

**心跳检测流程**：
```
每秒执行：
1. 随机选择5个节点发送PING
2. 如果节点在cluster-node-timeout内没回复PONG
3. failure_count++
4. 如果failure_count超过阈值（默认3）
5. 该节点标记为疑似下线
```

**主观下线 vs 客观下线**：
```
节点A视角：
- 15秒没收到B的PONG
- failure_count: 0 → 1 → 2 → 3
- B的主观下线

客观下线判断：
- 节点A向其他所有主节点发送FAIL消息
- 如果超过半数主节点也认为B主观下线
- 集群宣布B客观下线
- 触发故障转移
```

**实现要点**：
1. 节点心跳检测（PING/PONG）
2. 主观下线判断（节点认为另一个节点故障）
3. 客观下线判断（多个节点认为某个节点故障）
4. 故障传播

**心跳检测**：
```
1. 每秒随机选择节点发送PING
2. 节点收到PING后回复PONG
3. 超时未回复则failure_count++
4. failure_count超过阈值则认为节点下线
```

**下线判断**：
```
主观下线：
- 节点在cluster-node-timeout内未收到PONG
- failure_count > replica-validity-factor

客观下线：
- 集群中超过半数的主节点认为某节点主观下线
- 从节点也能参与投票
```

**验收标准**：
- 故障节点能被正确检测
- 主观下线和客观下线正确判断

### 任务5.2：从节点选举

**目标**：实现从节点选举为新主节点

**为什么需要这个组件？**

主节点故障后，需要从节点通过选举机制升级为新主节点：

1. **Raft风格选举**：从节点发起选举，获得多数票当选
2. **数据最新优先**：复制偏移量最新的从节点优先
3. **epoch机制**：防止重复投票和无效选举
4. **自动进行**：故障转移自动触发，不需要人工干预

**选举条件**：
```
不是所有从节点都能发起选举，必须满足：

1. 从节点的主节点处于客观下线状态
   （确保主节点真的故障了）

2. 从节点复制偏移量足够新
   （数据不能比主节点落后太多）

3. 从节点未执行其他故障转移
   （避免多个从节点同时选举）
```

**Raft风格选举流程**：
```
从节点C视角（主节点A的从节点）：

1. 等待A客观下线
   （确认主节点真的故障了）

2. 增加currentEpoch（选举轮次）
   currentEpoch = A的currentEpoch + 1

3. 发送FAILOVER_AUTH_REQUEST给所有主节点
   消息包含：
   - 我的currentEpoch
   - 我的复制偏移量
   - 我主节点的runid

4. 主节点收到后：
   - 检查是否投过票（每个epoch只能投一次）
   - 检查从节点数据是否够新
   - 满足条件则返回FAILOVER_AUTH_ACK

5. 从节点等待多数主节点的投票
   多数 = 主节点总数/2 + 1
   例如：3个主节点 → 需要2票

6. 获得多数票后：
   - 成为新主节点
   - 开始接收写请求
   - 向集群广播PONG消息
```

**为什么需要epoch？**
```
避免重复投票和无效选举：

旧epoch（可能来自网络分区）：
- 节点认为A故障，发送投票请求
- 但A实际上还活着，只是网络慢

新epoch：
- A恢复后有新epoch
- 旧epoch的投票无效
- 确保选举结果是最新的
```

**实现要点**：
1. 从节点发起选举（满足条件）
2. 主节点投票
3. 获得多数票的从节点成为新主节点
4. 向集群广播新主节点信息

**选举条件**：
```
1. 从节点的主节点处于客观下线状态
2. 从节点复制偏移量足够新
3. 从节点未执行其他故障转移
```

**选举流程（Raft风格）**：
```
1. 从节点增加currentEpoch，发送FAILOVER_AUTH_REQUEST
2. 主节点回复FAILOVER_AUTH_ACK（每个epoch只能投票一次）
3. 从节点等待多数票
4. 获得多数票后成为新主节点
5. 新主节点开始接收请求
6. 通知集群其他节点
```

**验收标准**：
- 从节点能正确发起选举
- 选举过程正确，多数票生效

### 任务5.3：故障转移执行

**目标**：执行故障转移，更新集群状态

**为什么需要这个组件？**

故障转移执行是将集群从"有节点故障"状态恢复到"正常"状态的关键步骤：

1. **从变主**：从节点升级为新的主节点
2. **槽接管**：新主节点接管原主节点的槽
3. **拓扑更新**：所有节点更新集群拓扑
4. **旧主处理**：原主节点恢复后变成从节点

**故障转移执行流程**：
```
场景：主节点A故障，从节点C当选新主节点

步骤1：C成为新主节点
- C向所有节点广播PONG消息
- 消息包含：A的槽现在归C管

步骤2：其他节点更新拓扑
- 所有节点收到C的PONG
- 更新槽映射：A的槽 → C
- 更新节点信息：C角色：MASTER

步骤3：C开始接收请求
- C现在负责A的所有槽
- 可以接收客户端的读写请求

步骤4：A恢复后处理
- A重新上线，发送PING
- 其他节点告诉A：C现在是主节点
- A变成C的从节点，开始复制C的数据
```

**故障转移前后对比**：
```
故障转移前：
┌───────┐         ┌───────┐
│  A(M) │ ──复制─→ │  C(R) │
│槽0-500│         │ 数据副本│
└───────┘         └───────┘

故障转移后：
┌───────┐         ┌───────┐
│  A(R) │ ←─复制── │  C(M) │
│       │          │槽0-500│
└───────┘          └───────┘
  变成从节点        升级为主节点
```

**为什么旧主节点要变成从节点？**
```
1. 保留数据：
   - A可能还有未复制给C的数据
   - 变成从节点后可以继续同步

2. 资源利用：
   - A的硬件资源（内存、CPU）还有用
   - 可以作为C的从节点提供备份

3. 自动恢复：
   - 不需要人工干预
   - 系统自动恢复到稳定状态
```

**实现要点**：
1. 从节点升级为主节点
2. 更新槽分配信息
3. 向集群广播PONG消息
4. 旧主节点重新加入时成为从节点

**故障转移后处理**：
```
1. 新主节点：接管原主节点的槽
2. 新主节点：开始接收请求
3. 其他节点：更新拓扑，指向新主节点
4. 旧主节点：重新加入后成为新主节点的从节点
```

**验收标准**：
- 故障转移后新主节点正常工作
- 槽接管正确
- 数据不丢失

### 任务5.4：网络分区处理

**目标**：处理网络分区场景

**为什么需要这个组件？**

网络分区（脑裂）是分布式系统中最难处理的问题之一：

1. **分区识别**：检测出集群被分割成多个子集群
2. **独立工作**：各子集群独立处理请求
3. **分区恢复**：网络恢复后合并子集群
4. **冲突处理**：处理分区期间的数据冲突

**网络分区场景**：
```
正常状态：
┌─────────────────────────────────────────┐
│              完整集群（3节点）             │
│  A(M) ←复制→ B(R) ←复制→ C(R)          │
└─────────────────────────────────────────┘

网络分区（假设A和B、C断开）：
┌──────────────────┐   ┌──────────────────┐
│    子集群1       │   │    子集群2       │
│   ┌─────────┐    │   │   ┌─────────┐    │
│   │  A(M)   │    │   │   │  B(R)   │    │
│   │ 独立工作 │    │   │   │ 独立工作 │    │
│   └─────────┘    │   │   └─────────┘    │
│                  │   │   ┌─────────┐    │
│                  │   │   │  C(R)   │    │
│                  │   │   │ 独立工作 │    │
└──────────────────┘   └──────────────────┘
```

**分区期间的问题**：
```
子集群1（A单独）：
- A认为B、C故障（客观下线）
- A继续服务，接收写请求
- 但B、C实际还活着

子集群2（B和C）：
- B、C互相认为对方存活
- B认为A故障，尝试故障转移
- B升级为主节点

分区恢复后：
┌─────────────────────────────────────────┐
│              合并后的集群（3节点）         │
│  B(M) ←复制→ C(R) ←复制→ A(R)          │
└─────────────────────────────────────────┘

问题：
- A在分区期间有写操作
- B在分区期间也有写操作
- 数据冲突如何处理？
```

**常见的分区处理策略**：
| 策略 | 原理 | 优点 | 缺点 |
|-----|------|-----|------|
| **最后写优先** | 以时间戳为准，保留最新数据 | 简单 | 可能丢失数据 |
| **主节点优先** | 分区前的主节点数据优先 | 符合预期 | 可能丢失数据 |
| **多数派优先** | 多数节点的分支优先 | 数据最完整 | 可能丢失少数派数据 |

**为什么需要特殊处理？**
```
普通分布式数据库也有分区问题，但Redis Cluster特点：
- 数据分片：每个节点只负责部分槽
- 主从复制：同一份数据有多份副本
- 故障转移：从节点可以升级

这使得分区问题更复杂：
- 不仅要处理写入冲突
- 还要处理槽映射不一致
- 还要处理主从角色冲突
```

**实现要点**：
1. 分区期间各子集群独立工作
2. 分区恢复后合并
3. 冲突检测和处理

**验收标准**：
- 分区期间各子集权能正常工作
- 分区恢复后能正确合并

### 任务6.2：槽迁移命令

**目标**：实现槽迁移管理命令

**为什么需要这些命令？**

槽迁移是集群扩缩容的核心，需要精细的控制命令：

1. **开始迁出**：标记某个槽要迁出到目标节点
2. **开始迁入**：标记某个槽要从源节点迁入
3. **取消迁移**：放弃正在进行的迁移
4. **完成迁移**：确认迁移完成，更新槽归属

**槽迁移状态转换**：
```
                    ┌─────────────────┐
                    │                 │
                    ↓                 │
        ┌───────────────────┐         │
        │     正常状态        │←────────────────┐
        │  (slotsAssigned)   │                │
        └───────────────────┘                │
                    │                         │
                    │ CLUSTER SETSLOT         │
                    │ <slot> MIGRATING       │
                    ↓                         │
        ┌───────────────────┐         ┌─────┴─────────┐
        │     迁出中状态       │────────→│ 迁入中状态   │
        │ (slotsMigrating)   │         │(slotsImporting)│
        └───────────────────┘         └───────────────┘
                    │                         │
                    │ CLUSTER SETSLOT         │
                    │ <slot> STABLE          │
                    ↓                         │
        ┌───────────────────┐         ┌─────┴─────────┐
        │     正常状态        │←────────│ 完成迁移      │
        │                    │         │              │
        └───────────────────┘         └──────────────┘
```

**迁移命令详解**：
| 命令 | 作用 | 执行节点 |
|-----|------|---------|
| SETSLOT MIGRATING | 标记槽为迁出状态 | 源节点 |
| SETSLOT IMPORTING | 标记槽为迁入状态 | 目标节点 |
| SETSLOT STABLE | 取消迁移 | 任一节点 |
| SETSLOT node_id | 完成迁移 | 任一节点 |

**使用示例**：
```bash
# Node1执行：开始将槽12345迁出到Node2
CLUSTER SETSLOT 12345 MIGRATING 192.168.1.20:6379

# Node2执行：开始将槽12345迁入
CLUSTER SETSLOT 12345 IMPORTING 192.168.1.10:6379

# 迁移完成后，任一节点执行：
CLUSTER SETSLOT 12345 192.168.1.20:6379

# 或者取消迁移：
CLUSTER SETSLOT 12345 STABLE
```

**实现要点**：
1. CLUSTER SETSLOT <slot> MIGRATING <node_id>命令
2. CLUSTER SETSLOT <slot> IMPORTING <node_id>命令
3. CLUSTER SETSLOT <slot> STABLE命令
4. CLUSTER SETSLOT <slot> <node_id>命令
```
CLUSTER SETSLOT <slot> MIGRATING <node_id>   # 开始迁出
CLUSTER SETSLOT <slot> IMPORTING <node_id>  # 开始迁入
CLUSTER SETSLOT <slot> STABLE               # 取消迁移
CLUSTER SETSLOT <slot> <node_id>             # 完成迁移
```

**验收标准**：

- 迁移状态正确转换

### 任务6.3：节点管理

**目标**：实现节点生命周期管理

**为什么需要这个组件？**

节点的启动、运行、关闭、重启都需要正确管理：

1. **启动初始化**：节点启动时加载配置、建立连接
2. **配置持久化**：节点配置信息保存到磁盘
3. **优雅关闭**：处理完当前请求后再退出
4. **故障恢复**：节点重启后能正确恢复状态

**节点启动流程**：
```
节点启动：
1. 读取配置文件（cluster-enabled、节点ID等）
2. 检查nodes.conf是否存在？
   - 存在 → 加载节点信息，恢复拓扑
   - 不存在 → 生成新节点ID，成为独立节点
3. 如果配置了replicaof → 尝试连接主节点，建立复制
4. 如果cluster-enabled=yes → 进入集群模式
5. 开始接受客户端连接和Gossip消息
```

**为什么需要节点信息持久化？**
```
节点重启后：
- 如果不持久化 → 所有节点信息丢失
- 节点变成新节点 → 需要重新加入集群
- 之前的角色、槽分配都丢失

持久化后：
- 节点ID保持不变
- 槽分配信息保留
- 可以快速恢复集群拓扑
```

**优雅关闭的重要性**：
```
不优雅关闭：
- 节点突然断开
- 其他节点认为节点故障
- 可能触发故障转移
- 需要重新全量复制

优雅关闭：
- 节点通知其他节点我要关闭
- 其他节点更新拓扑
- 不触发故障转移
- 节点信息保留
```

**实现要点**：
1. 节点启动和初始化
2. 节点信息持久化
3. 节点优雅关闭
4. 节点重启动恢复
1. 节点启动和初始化
2. 节点信息持久化
3. 节点优雅关闭
4. 节点重启动恢复

**验收标准**：
- 节点能正确启动和关闭
- 节点信息能正确持久化和恢复

## 常见问题与解决方案

1. **内存泄漏问题**：使用valgrind工具检测内存泄漏，重点检查内存池和哈希表节点的释放
2. **多线程竞态条件**：使用ThreadSanitizer工具检测数据竞争，确保所有共享资源都被正确保护
3. **性能瓶颈**：使用perf工具分析CPU和内存使用情况，找到热点代码进行优化
4. **网络问题**：使用Wireshark抓包分析网络流量，排查协议解析和数据收发问题
5. **数据一致性问题**：使用redis-cli --latency测试延迟，redis-benchmark测试吞吐量
6. **集群脑裂问题**：网络分区时各子集群独立工作，分区恢复后按配置策略合并
