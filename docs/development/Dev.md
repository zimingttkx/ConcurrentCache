# ConcurrentCache 开发路线指导（完整版）

## 项目开发总览

本项目采用**分阶段、增量式**的开发方式，将整个项目分为4个大阶段，每个阶段又分为多个小任务。每个任务都有明确的**目标**、**实现要点**、**测试方法**和**验收标准**。

**总开发周期**：约12-16周（每周投入20-30小时）

**版本规划**：
- **Version1（骨架版本）**：项目最小可运行版本，约2-3周
- **Version2（基础版本）**：完整基础功能版，约4-6周
- **Version3（增强版本）**：完整功能和持久化版，约8-11周
- **Version4（分布式版本）**：集群功能版（可选），约12-16周

## 章节2：内存池实现（第3-4周）

### 任务2.1：ThreadCache实现

**目标**：实现线程本地缓存层

**实现要点**：
1. 使用线程本地存储（TLS）
2. 按大小分类管理内存对象（Size Class：8B、16B、32B...256KB）
3. 无锁分配释放
4. 线程退出时归还内存到CentralCache

**Size Class设计**：
- 8B, 16B, 32B, 48B, 64B, 96B, 128B, 192B, 256B, 384B, 512B, 768B, 1024B, 1536B, 2048B, 3072B, 4096B, 6144B, 8192B, 12288B, 16384B, 24576B, 32768B, 49152B, 65536B, 98304B, 131072B, 196608B, 262144B

**验收标准**：
- ThreadCache分配和释放无锁
- 多线程环境下不会相互干扰

---

### 任务2.2：CentralCache实现

**目标**：实现中心缓存层

**实现要点**：
1. 每个Size Class对应一个Span链表
2. 使用细粒度锁（每个Span链表一把锁）
3. 从PageCache申请内存页
4. 向ThreadCache分配内存对象
5. 回收ThreadCache归还的内存

**Span结构**：
- 管理一组连续内存页
- 记录属于哪个Size Class
- 记录空闲对象数量

**验收标准**：
- CentralCache能正确管理多个Span链表
- 细粒度锁能正确保护各个链表

---

### 任务2.3：PageCache实现

**目标**：实现页缓存层

**实现要点**：
1. 以4KB页为单位管理内存
2. 管理Span结构
3. 向系统申请和释放内存
4. 合并相邻的空闲Span

**验收标准**：
- PageCache能正确管理内存页
- 能正确向系统申请和释放内存

---

### 任务2.4：内存池整合

**目标**：整合三级缓存，实现完整的内存池

**实现要点**：
1. ThreadCache向CentralCache申请内存
2. CentralCache向PageCache申请页
3. ThreadCache空闲对象过多时归还给CentralCache
4. CentralCache空闲Span过多时归还给PageCache
5. 实现大内存分配（>256KB）直接使用malloc

**验收标准**：
- 内存分配和释放在单线程和多线程环境下都正确
- 内存池性能优于原生malloc/free（多线程环境下至少快3倍）
- 内存碎片率低于10%

## 章节5：缓存核心实现（第7-8周）

### 任务5.1：分段锁哈希表

**目标**：实现高性能的分段锁全局哈希表

**实现要点**：
1. 将哈希表划分为多个独立的分片（默认16个，可配置）
2. 每个分片有自己的读写锁
3. 每个分片使用std::unordered_map存储键值对
4. 键的哈希值对分片数取模，确定键属于哪个分片
5. 实现基本操作：get、set、del、exists
6. 使用内存池分配哈希表节点内存

**验收标准**：
- 所有基本操作在单线程和多线程环境下都正确
- 并发读写性能明显优于全局锁哈希表（至少快5倍）

---

### 任务5.2：过期字典和过期删除

**目标**：实现键过期自动删除功能

**实现要点**：
1. 每个分片维护一个过期字典，存储键到过期时间的映射
2. 过期时间使用毫秒级时间戳
3. 实现惰性删除：访问键时检查是否过期，如果过期则删除
4. 实现定期删除：每隔100ms，随机抽取20个键检查是否过期
5. 限制每次定期删除的时间不超过25ms
6. 实现EXPIRE、TTL、PERSIST命令

**验收标准**：
- 过期键能被正确删除
- EXPIRE、TTL、PERSIST命令工作正常
- 定期删除不会占用过多CPU

---

### 任务5.3：LRU缓存替换算法

**目标**：实现LRU缓存替换算法

**实现要点**：
1. 定义EvictionPolicy基类，所有替换算法都继承自该类
2. 每个分片维护自己的LRU链表
3. 访问键时更新访问时间戳，将节点移到LRU链表头部
4. 当内存使用达到上限时，从LRU链表尾部开始淘汰
5. 实现近似LRU算法：随机抽取5个键，淘汰其中最久未访问的
6. 实现CONFIG SET maxmemory和CONFIG GET maxmemory命令

**验收标准**：
- 内存使用能被控制在配置的上限以内
- 最久未访问的键优先被淘汰
- 近似LRU算法的准确性达到90%以上

## 章节1：哈希槽分片

### 任务1.1：槽映射

**目标**：实现Redis Cluster风格的哈希槽分片

**实现要点**：

3. **性能瓶颈**：使用perf工具分析CPU和内存使用情况，找到热点代码进行优化
4. **网络问题**：使用Wireshark抓包分析网络流量，排查协议解析和数据收发问题
