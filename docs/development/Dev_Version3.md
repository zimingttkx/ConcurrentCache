# ConcurrentCache 增强版本（Version3）

## 版本概述

**目标**：添加更多数据类型和持久化功能，并进行全面的性能优化。

增强版本是项目的功能完备版本，实现所有Redis兼容的数据类型和持久化机制。

**预计开发周期**：8-11周

**核心功能**：
- 哈希(Hash)数据类型
- 列表(List)数据类型
- 集合(Set)数据类型
- 有序集合(ZSet)数据类型
- LFU、FIFO、Random缓存替换算法
- RDB快照持久化
- AOF追加日志持久化
- 性能优化（锁优化、I/O优化、内存优化）

## 组件作用说明

### 为什么要支持更多数据类型？

在Version1和Version2中，我们只支持简单的String（字符串）类型。但实际应用场景中，数据结构的需求是多种多样的：

| 数据类型 | 解决的问题 | 典型使用场景 |
|---------|-----------|-------------|
| **Hash（哈希）** | 结构化数据存储 | 用户信息、商品详情、配置项 |
| **List（列表）** | 有序队列、消息流 | 消息队列、最新消息、时间线 |
| **Set（集合）** | 去重、集合运算 | 标签系统、好友关系、共同好友 |
| **ZSet（有序集合）** | 排行榜、权重排序 | 游戏积分榜、商品热度、优先级队列 |

### 为什么要支持多种缓存替换算法？

不同的业务场景需要不同的淘汰策略：

| 算法 | 淘汰依据 | 适用场景 |
|-----|---------|---------|
| **LRU** | 最近最少使用 | 热点数据缓存、页面缓存 |
| **LFU** | 最不经常使用 | 高频访问数据、API响应缓存 |
| **FIFO** | 先进先出 | 临时数据、会话缓存 |
| **Random** | 随机淘汰 | 负载均衡、简单缓存场景 |

### 为什么要持久化？

内存缓存的最大问题就是**数据易失**——服务器重启、断电等情况会导致所有数据丢失。持久化将内存数据保存到磁盘，确保数据安全。

| 持久化方式 | 特点 | 优缺点 |
|-----------|------|--------|
| **RDB** | 定时快照 | 恢复快、文件小，但可能丢失最后一次快照后的数据 |
| **AOF** | 追加日志 | 数据安全性高，但文件较大 |

## 章节1：更多数据类型（第1-3周）

### 任务1.1：哈希数据类型

**目标**：实现哈希数据类型

**为什么需要这个组件？**

哈希数据类型（Hash）是Redis中最常用的数据结构之一，它解决了以下问题：

1. **结构化存储**：将一个对象的多个属性存储在一个键下，避免为每个属性创建单独的键
2. **减少键数量**：100个用户的用户名信息，用String类型需要100个键（user:1:name, user:2:name...），用Hash只需要1个键（users是一个Hash，field是用户ID）
3. **原子操作**：可以单独更新某个field，而不影响其他fields
4. **内存效率**：相比为每个属性存储单独的键，Hash更节省内存

**使用场景示例**：
- 用户信息存储：user:1000 → {name: "张三", age: 25, city: "北京"}
- 商品详情：product:SKU123 → {name: "iPhone15", price: 6999, stock: 100}
- 配置项存储：config:redis → {maxmemory: "4gb", port: 6379}

**实现要点**：
1. 定义Hash类，存储哈希表数据
2. 内部使用std::unordered_map存储field-value对
3. 每个分片的哈希表独立管理

**命令列表**：
```
HSET key field value          # 设置哈希字段
HGET key field                # 获取哈希字段值
HDEL key field [field ...]    # 删除哈希字段
HEXISTS key field             # 检查字段是否存在
HGETALL key                   # 获取所有字段和值
HKEYS key                     # 获取所有字段名
HVALS key                     # 获取所有值
HLEN key                      # 获取字段数量
HINCRBY key field increment   # 字段值增加
HSETNX key field value        # 字段不存在时设置
```

**验收标准**：
- 所有哈希命令正确执行
- 与redis-cli兼容
- 边界情况处理正确（field不存在、类型错误等）

### 任务1.2：列表数据类型

**目标**：实现列表数据类型

**为什么需要这个组件？**

列表数据类型（List）是常用的有序数据结构，它解决了以下问题：

1. **有序存储**：元素按插入顺序排列，可以按索引访问
2. **两端操作**：支持从头部（LPUSH/LPOP）和尾部（RPUSH/RPOP）高效操作
3. **消息队列**：可以实现简单的消息队列系统
4. **最新N个元素**：快速获取最新的N条记录（如最新消息、时间线）

**使用场景示例**：
- 消息队列：LPUSH将消息放入队列，RPOP取出消息（先进先出）
- 最新消息列表：LPUSH将新消息插入列表，LRANGE获取最新10条
- 任务队列：LPUSH添加任务，BRPOP阻塞等待任务（生产者-消费者模式）
- 关注列表：按时间顺序存储用户关注的人的动态

**实现要点**：
1. 定义List类，存储列表数据
2. 内部使用双向链表或压缩列表实现
3. 支持从头部或尾部操作

**命令列表**：
```
LPUSH key value [value ...]   # 从左侧插入
RPUSH key value [value ...]   # 从右侧插入
LPOP key                     # 从左侧弹出
RPOP key                     # 从右侧弹出
LLEN key                     # 获取列表长度
LRANGE key start stop        # 获取范围内元素
LREM key count value         # 移除指定元素
LSET key index value         # 设置指定位置元素
LINDEX key index             # 获取指定位置元素
LTRIM key start stop         # 裁剪列表
```

**验收标准**：
- 所有列表命令正确执行
- 与redis-cli兼容
- LPUSH/RPUSH能正确插入多个元素
- LRANGE负索引正确处理

### 任务1.3：集合数据类型

**目标**：实现集合数据类型

**为什么需要这个组件？**

集合数据类型（Set）是存储无序且不重复元素的数据结构，它解决了以下问题：

1. **自动去重**：同一个元素只能出现一次，适合需要唯一性的场景
2. **高效判断存在**：O(1)时间复杂度判断元素是否在集合中
3. **集合运算**：支持交集（SINTER）、并集（SUNION）、差集（SDIFF）等操作
4. **标签系统**：方便地添加、删除、查询标签

**使用场景示例**：
- 标签系统：article:100的标签 → {"技术", "Redis", "高性能"}，快速判断某标签是否存在
- 好友关系：user:1000的粉丝 → {"1001", "1002", "1003"}，快速添加/删除粉丝
- 共同好友：计算用户A和用户B的共同粉丝数量（SINTER）
- IP访问记录：记录访问服务器的IP列表，自动去重，统计UV
- 推荐系统：用户兴趣标签集合，进行协同过滤

**实现要点**：
1. 定义Set类，存储集合数据
2. 内部使用std::unordered_set存储
3. 支持集合运算

**命令列表**：
```
SADD key member [member ...]  # 添加成员
SREM key member [member ...] # 移除成员
SISMEMBER key member          # 检查成员是否存在
SMEMBERS key                  # 获取所有成员
SCARD key                     # 获取成员数量
SPOP key [count]              # 随机弹出成员
SRANDMEMBER key [count]       # 随机获取成员
SINTER key [key ...]          # 交集
SUNION key [key ...]          # 并集
SDIFF key [key ...]           # 差集
```

**验收标准**：
- 所有集合命令正确执行
- 与redis-cli兼容
- 集合运算结果正确

### 任务1.4：有序集合数据类型

**目标**：实现有序集合数据类型

**为什么需要这个组件？**

有序集合数据类型（ZSet）是每个元素都有一个分数（score）与之关联的数据结构，元素按分数排序。它解决了以下问题：

1. **排序需求**：需要按某个数值（分数）对数据进行排序
2. **排名查询**：快速获取元素的排名（正序/倒序）
3. **范围查询**：按分数范围查询元素
4. **权重系统**：实现带权重的优先级队列

**使用场景示例**：
- 游戏排行榜：players → {(score: 10000, name: "张三"), (score: 9500, name: "李四")}
  - ZREVRANGE获取前10名玩家
  - ZRANK获取某玩家的排名
  - ZINCRBY增加玩家积分
- 商品热度榜：products:hot → {(score: 1000, item: "iPhone15"), (score: 800, item: "MacBook")}
- 优先级队列：tasks:priority → {(score: 10, task: "紧急任务"), (score: 5, task: "普通任务")}
- 时间线排序：timeline:user100 → {(score: timestamp, event: "发布文章")}，按时间排序

**实现要点**：
1. 定义ZSet类，存储有序集合数据
2. 内部使用std::map或跳表实现（按分数排序）
3. 支持按分数或索引访问

**命令列表**：
```
ZADD key score member [score member ...]  # 添加成员
ZREM key member [member ...]             # 移除成员
ZSCORE key member                        # 获取成员分数
ZRANGE key start stop [WITHSCORES]       # 按索引范围获取
ZREVRANGE key start stop [WITHSCORES]   # 反向按索引范围获取
ZRANK key member                         # 获取成员排名
ZREVRANK key member                      # 反向获取成员排名
ZCARD key                                # 获取成员数量
ZCOUNT key min max                       # 统计分数范围内的成员
ZINCRBY key increment member             # 增加成员分数
ZRANGEBYSCORE key min max [WITHSCORES]   # 按分数范围获取
```

**验收标准**：
- 所有有序集合命令正确执行
- 与redis-cli兼容
- 成员按分数正确排序
- 分数相同时按字典序排序

## 章节2：更多缓存替换算法（第4周）

### 任务2.1：LFU算法

**目标**：实现LFU（最不经常使用）缓存替换算法

**为什么需要这个组件？**

LFU（Least Frequently Used）算法根据数据的访问频率进行淘汰，解决了以下问题：

1. **高频数据优先保留**：访问频率高的数据更可能是热点数据，应该保留在缓存中
2. **区分冷热数据**：能够区分访问次数差异明显的数据
3. **适用持续热点场景**：某些数据被频繁访问，即使最近没有被访问，也应该保留

**与LRU的区别**：
- LRU：看"最近"，如果数据刚刚被访问过就保留
- LFU：看"频率"，如果数据被访问过很多次就保留

**使用场景示例**：
- API响应缓存：某些API被高频调用，LFU确保这些API响应保留在缓存
- 热门商品缓存：热门商品页面被频繁访问，LFU保留这些数据
- 配置缓存：某些配置项被反复读取，LFU避免频繁重新解析

**实现要点**：
1. 每个键维护访问计数器（counter）
2. 每个键维护访问时间戳（last_access_time）
3. 淘汰时选择计数器最小的键
4. 计数器有衰减机制（避免计数器无限增长）
5. 淘汰时遇到计数器相同的键，选择时间更早的

**类设计**：
```cpp
class LFUEvictionPolicy : public EvictionPolicy {
private:
    std::unordered_map<std::string, CounterInfo> infos_;
    // CounterInfo包含：counter_, last_access_time_

    void on_access(const std::string& key) override;
    std::string evict() override;
};
```

**验收标准**：
- LFU算法正确工作
- 频繁访问的键不会被淘汰
- 计数器衰减机制正常

### 任务2.2：FIFO算法

**目标**：实现FIFO（先进先出）缓存替换算法

**为什么需要这个组件？**

FIFO（First In First Out）算法按照数据进入缓存的先后顺序进行淘汰，解决了以下问题：

1. **简单实现**：不需要维护复杂的访问信息，实现简单、性能开销小
2. **公平性**：保证每个数据都有平等的被淘汰机会
3. **适合短期数据**：适合存储临时数据、session数据等

**使用场景示例**：
- 临时缓存：存储计算中间结果，结果只需要保留一轮计算
- 会话缓存：用户会话数据，会话过期后按顺序淘汰
- 限流窗口：存储最近的请求记录，按时间顺序清理
- 简单队列场景：不需要考虑访问频率，只需要按顺序处理

**实现要点**：
1. 每个键维护插入时间戳
2. 淘汰时选择最早插入的键
3. 使用队列或链表维护插入顺序

**类设计**：
```cpp
class FIFOEvictionPolicy : public EvictionPolicy {
private:
    std::list<std::string> fifo_list_;
    std::unordered_map<std::string, std::list<std::string>::iterator> pos_map_;

    void on_access(const std::string& key) override;  // FIFO中访问不改变顺序
    std::string evict() override;
};
```

**验收标准**：
- FIFO算法正确工作
- 最早插入的键优先被淘汰

### 任务2.3：Random算法

**目标**：实现Random（随机）缓存替换算法

**为什么需要这个组件？**

Random算法随机选择一个键进行淘汰，是最简单的淘汰策略，解决了以下问题：

1. **零开销**：不需要维护任何额外信息，实现极其简单
2. **无偏好性**：对所有数据一视同仁，避免某些数据长期占用缓存
3. **负载均衡**：在某些场景下，随机淘汰也能达到不错的效果

**使用场景示例**：
- 简单缓存场景：不需要考虑访问模式，只需保证缓存不溢出
- 基准测试：作为其他淘汰算法的基准对比
- 分发场景：随机选择数据进行处理（如随机负载均衡）
- 防御性缓存：防止缓存被特定数据占满

**实现要点**：
1. 随机选择一个键淘汰
2. 实现简单，性能最高

**类设计**：
```cpp
class RandomEvictionPolicy : public EvictionPolicy {
private:
    std::vector<std::string> keys_;  // 所有键的列表

    void on_access(const std::string& key) override;
    std::string evict() override;
};
```

**验收标准**：
- Random算法正确工作
- 每次淘汰随机选择

### 任务2.4：动态切换

**目标**：支持动态切换替换算法

**为什么需要这个组件？**

动态切换允许在运行时改变缓存淘汰策略，解决了以下问题：

1. **灵活适配**：不同业务场景需要不同策略，无需重启服务
2. **A/B测试**：可以测试不同策略的效果
3. **渐进式迁移**：从一种策略平滑过渡到另一种策略
4. **运维友好**：线上问题可以快速调整策略

**支持的可配置策略**：
| 策略 | 描述 | 适用场景 |
|-----|------|---------|
| volatile-lru | 从已设置过期时间的键中淘汰LRU | 混合数据，有明确冷热区分 |
| allkeys-lru | 从所有键中淘汰LRU | 全部数据都是缓存，无明确过期 |
| volatile-lfu | 从已设置过期时间的键中淘汰LFU | 高频访问数据带过期时间 |
| allkeys-lfu | 从所有键中淘汰LFU | 高频访问数据无过期时间 |
| volatile-fifo | 从已设置过期时间的键中淘汰FIFO | 临时数据有过期时间 |
| allkeys-fifo | 从所有键中淘汰FIFO | 全部数据都是临时的 |
| volatile-random | 从已设置过期时间的键中随机淘汰 | 无明显访问规律 |
| allkeys-random | 从所有键中随机淘汰 | 无明显访问规律 |
| noeviction | 不淘汰，返回错误 | 不允许淘汰，严格控制内存 |

**实现要点**：
1. 实现CONFIG SET maxmemory-policy命令
2. 支持动态切换替换策略
3. 切换时保留已有键的访问信息

**支持的策略**：
```
volatile-lru    # 从已设置过期时间的键中淘汰LRU
allkeys-lru     # 从所有键中淘汰LRU
volatile-lfu    # 从已设置过期时间的键中淘汰LFU
allkeys-lfu     # 从所有键中淘汰LFU
volatile-fifo   # 从已设置过期时间的键中淘汰FIFO
allkeys-fifo    # 从所有键中淘汰FIFO
volatile-random # 从已设置过期时间的键中随机淘汰
allkeys-random  # 从所有键中随机淘汰
noeviction      # 不淘汰，返回错误
```

**验收标准**：
- 能动态切换替换算法
- 切换过程不影响服务

## 章节3：RDB持久化（第5-6周）

### 任务3.1：RDB文件生成

**目标**：实现RDB快照持久化

**为什么需要这个组件？**

RDB（Redis Database）快照持久化将内存数据以二进制格式保存到磁盘，解决了以下问题：

1. **数据备份**：定期保存数据快照，防止数据丢失
2. **快速恢复**：重启时直接加载RDB文件，恢复速度快（相比AOF）
3. **文件紧凑**：二进制格式，文件体积小，便于传输和备份
4. **灾备恢复**：可以定期备份到远程服务器

**使用场景示例**：
- 定时备份：每天凌晨3点执行BGSAVE，将数据备份
- 增量备份：每小时执行一次，保留最近24小时的快照
- 异地灾备：将RDB文件同步到异地数据中心
- 主从同步：主节点生成RDB，通过网络传输给从节点

**RDB文件的优缺点**：

| 优点 | 缺点 |
|-----|------|
| 文件紧凑，节省存储空间 | 可能丢失最后一次快照后的数据 |
| 恢复速度快 | 需要fork进程，大数据量时可能阻塞 |
| 适合备份和灾难恢复 | 无法实现实时持久化 |

**实现要点**：
1. 实现SAVE命令：主线程阻塞保存快照
2. 实现BGSAVE命令：fork子进程保存快照，父进程继续处理请求
3. 实现RDB文件格式（按Redis RDB格式）
4. 实现自动保存（根据配置文件中的save条件）
5. BGSAVE期间拒绝新的BGSAVE请求

**RDB文件格式**：
```
+------+--------------------------------------------------+
| MAGIC |              RDB文件标识 "REDIS0009"            |
+------+--------------------------------------------------+
| 辅助字段 | version, aux_field等                         |
+------+--------------------------------------------------+
| 数据库 | SELECTDB + 数据库编号 + 键值对数据             |
+------+--------------------------------------------------+

```

```

+------+------+------+------+------+------+------+------+
```

**类设计**：

## 章节4：AOF持久化（第7-8周）

### 任务4.1：AOF文件写入

**目标**：实现AOF追加日志持久化

**为什么需要这个组件？**

AOF（Append Only File）追加日志持久化将每个写命令记录到文件，解决了以下问题：

1. **数据安全性更高**：可以配置每个写操作都刷盘，最大限度保证不丢数据
2. **实时持久化**：相比RDB的定时快照，AOF可以更实时地保存数据
3. **命令可读**：AOF文件是文本格式，可以直接查看和编辑
4. **日志价值**：可以用于分析操作历史、审计等

**RDB vs AOF对比**：

| 特性 | RDB | AOF |
|-----|-----|-----|
| 文件大小 | 小（二进制） | 大（文本命令） |
| 恢复速度 | 快 | 慢（需要重放命令） |
| 数据安全性 | 可能丢失快照后的数据 | 可配置（always/everysec/no） |
| 写入方式 | 定时快照 | 追加写入 |
| 优先级 | 启动时先检查 | 启动时优先加载 |

**刷盘策略**：
| 策略 | 说明 | 安全性 | 性能 |
|-----|------|-------|-----|
| always | 每个命令都刷盘 | 最高 | 最低 |
| everysec | 每秒刷盘一次 | 中等 | 较高 |
| no | 由操作系统决定 | 最低 | 最高 |

**实现要点**：
1. 实现AOF文件的创建和追加
2. 支持三种刷盘策略：always、everysec、no
3. 将每个写命令追加到AOF文件
4. 命令序列化使用RESP协议格式

**刷盘策略**：
```
always    # 每个命令都刷盘（最安全，性能最低）
everysec  # 每秒刷盘一次（折中方案）
no        # 由操作系统决定刷盘时机（性能最高，可能丢失最多数据）
```

**类设计**：
```cpp
class AOFPersistence {
public:
    void append_command(const std::vector<std::string>& args);
    void flush();

    void set_appendfsync(const std::string& mode);

private:
    std::string filename_;
    AppendFSyncMode mode_;  // always/everysec/no
    // ...
};
```

**验收标准**：
- AOF文件能正确生成和追加
- 三种刷盘策略都正常工作
- 数据能正确恢复

### 任务4.2：AOF重写

**目标**：实现AOF重写功能

**为什么需要这个组件？**

AOF重写解决AOF文件膨胀问题，减小文件体积，解决了以下问题：

1. **文件压缩**：删除冗余命令，生成最小的恢复命令集
2. **空间节省**：大文件占用大量磁盘空间，重写后可大幅减小
3. **恢复速度**：文件减小后，加载速度更快
4. **持续写入**：重写过程中继续处理新命令，不阻塞服务

**重写原理示例**：
```
原始AOF文件：
SET key1 v1    # 被后续命令覆盖
SET key1 v2    # 被后续命令覆盖
SET key1 v3    # 最新值，保留
DEL key2       # 被DEL后无意义
SET key2 new_value  # 有意义，保留

重写后AOF：
SET key1 v3
SET key2 new_value
```

**使用场景示例**：
- 定期压缩：每小时执行一次BGREWRITEAOF，控制文件大小
- 启动前优化：服务器启动前重写AOF，减小文件体积
- 磁盘空间不足时：紧急重写AOF释放空间

**实现要点**：
1. 当AOF文件过大时，重写AOF文件
2. 只保留能恢复当前数据的最小命令集
3. 实现BGREWRITEAOF命令
4. 重写过程中可以继续处理新命令

**重写策略**：
```
例如：
SET key1 v1
SET key1 v2
→ 只保留 SET key1 v2

DEL key1
SET key1 v1
→ 只保留 SET key1 v1
```

**类设计**：
```cpp
class AOFRewrite {
public:
    void bg_rewrite();
    bool is_rewriting() const;

private:
    void rewrite();  // 执行重写
    void rewrite_database(std::ostream& os, int db_id);
};
```

**验收标准**：
- AOF重写能有效减小文件大小
- 重写过程中不影响服务
- 重写后的AOF能正确恢复数据

### 任务4.3：混合持久化

**目标**：支持RDB和AOF混合持久化

**为什么需要这个组件？**

混合持久化结合RDB和AOF的优点，提供了最佳的数据安全性和恢复速度平衡：

1. **最优恢复**：同时存在时优先加载AOF（数据更完整）
2. **快速恢复**：先加载RDB作为基础，再重放AOF增量
3. **文件优化**：重写AOF时使用RDB格式作为前缀，文件更小
4. **兼容性好**：可以利用两种格式各自的优势

**加载优先级**：
```

3. 如果都不存在 → 从空状态启动
```

**使用场景示例**：
- 生产环境：同时开启RDB和AOF，获得双重数据保护
- 快速启动：RDB提供快速启动，AOF保证数据完整性
- 数据迁移：混合模式下可以灵活选择恢复方式

**实现要点**：
1. 服务器启动时优先加载AOF（数据更完整）
2. 如果AOF不存在则加载RDB
3. 重写AOF时生成RDB格式的前缀

**加载优先级**：
```
1. 如果AOF存在且非空，加载AOF
2. 否则如果RDB存在，加载RDB
3. 如果都不存在，从空状态启动
```

**验收标准**：
- 混合持久化功能正常
- AOF优先加载
- 数据完整性最高

## 章节5：性能优化（第9-10周）

### 任务5.1：锁优化

**目标**：减少锁竞争，提高并发性能

**为什么需要这个组件？**

在高并发场景下，锁竞争是性能瓶颈的主要原因之一。锁优化解决了以下问题：

1. **减少等待时间**：减少线程等待锁的时间
2. **提高吞吐量**：更多线程可以同时工作
3. **降低延迟**：请求处理时间更稳定

**锁竞争的原因**：
- 多个线程访问同一个共享资源
- 锁粒度太大（粗粒度锁）
- 读多写少场景使用互斥锁

**优化方向与效果**：

| 优化方法 | 解决的问题 | 预期效果 |
|---------|-----------|---------|
| 增加分片数 | 减少不同线程访问同一分片 | 锁竞争减少50%+ |
| 使用读写锁 | 读操作远多于写操作 | 读性能提升 |
| 原子操作 | 简单计数器更新 | 避免加锁开销 |
| 乐观锁 | 写冲突概率低 | 减少锁持有时间 |

**实现要点**：
1. 进一步细化锁粒度（减少临界区）
2. 使用无锁数据结构替换有锁数据结构
3. 优化读写锁的使用（读多写少场景）
4. 使用原子操作替代互斥锁（简单计数器等）

**优化方向**：
```
1. 哈希表分片数可配置（增加分片数减少锁竞争）
2. 使用std::atomic代替简单计数器
3. 使用乐观锁减少写锁持有时间
```

**验收标准**：
- 锁竞争明显减少
- 并发性能提升20%以上

### 任务5.2：内存优化

**目标**：优化内存使用，减少碎片

**为什么需要这个组件？**

内存优化确保缓存在有限的内存空间内存储更多数据，提高了内存使用效率：

1. **存储更多数据**：同样的内存可以缓存更多键值对
2. **降低内存成本**：减少服务器内存需求
3. **提高命中率**：更多数据在缓存中，命中率更高
4. **减少GC压力**：合理的内存布局减少内存碎片

**内存浪费的常见原因**：

| 问题 | 说明 | 解决方案 |
|-----|------|---------|
| 内存碎片 | 分配和释放导致空闲内存不连续 | 内存池、对象池 |
| 小字符串开销 | 每个字符串都有额外元数据 | 短字符串内部化（embstr） |
| 数据结构开销 | 列表/哈希/集合的内部结构 | 压缩列表（ziplist） |
| 内存对齐 | CPU对齐要求导致的浪费 | 调整数据结构布局 |

**使用场景示例**：
- embstr编码：存储小于32字节的字符串时，使用连续的内存块
- ziplist编码：列表/哈希/集合元素较少时，使用紧凑的zlplist格式
- 内存对齐：调整结构体成员顺序，减少内存浪费

**实现要点**：
1. 优化数据结构，减少内存开销
2. 实现内存对齐，提高CPU缓存命中率
3. 优化内存池，减少内存碎片
4. 压缩存储（短字符串内部化）

**优化方向**：
```
1. 字符串长度小于阈值时使用embstr编码
2. 列表使用ziplist编码（元素少时）
3. 哈希使用ziplist编码（field少时）
```

**验收标准**：
- 内存碎片率降低
- 内存使用率提升

### 任务5.3：I/O优化

**目标**：优化网络I/O和文件I/O性能

**为什么需要这个组件？**

I/O操作是系统的主要性能瓶颈之一，I/O优化解决了以下问题：

1. **减少系统调用**：每次系统调用都有上下文切换开销
2. **提高吞吐量**：批量操作比多次小操作效率高
3. **降低延迟**：减少数据在用户态和内核态之间的拷贝
4. **网络效率**：优化TCP参数提高网络传输效率

**I/O优化的核心问题**：

| 问题 | 原因 | 解决方案 |
|-----|------|---------|

| 网络延迟 | TCP握手、拥塞控制 | TCP参数优化 |

**使用场景示例**：

- 持久化优化：批量命令重放，减少IO次数

**实现要点**：

3. 使用零拷贝技术（sendfile）
4. TCP参数优化（tcp_nodelay、tcp_keepalive等）

**优化方向**：
```

2. 使用sendfile系统调用
3. 合并多个小写为一次write
```

**验收标准**：

### 任务5.4：CPU优化

**目标**：优化CPU使用，提高处理效率

**为什么需要这个组件？**

CPU优化让CPU更高效地处理请求，减少资源浪费：

1. **减少上下文切换**：线程在不同CPU核心间切换有开销
2. **提高缓存命中率**：CPU缓存命中率影响性能
3. **编译器优化**：让编译器生成更高效的机器码
4. **消除伪共享**：避免多核CPU缓存行失效

**CPU优化的核心问题**：

| 问题 | 原因 | 解决方案 |
|-----|------|---------|
| 上下文切换 | 线程在CPU核心间迁移 | CPU绑定（affinity） |
| 缓存未命中 | 数据不在CPU缓存中 | 数据对齐、预取 |
| 分支预测失败 | CPU分支预测错误 | likely/unlikely宏 |
| 伪共享 | 多核修改同一缓存行 | padding隔离 |

**使用场景示例**：
- SubReactor线程绑定：每个SubReactor绑定固定CPU核心，减少调度开销
- 热点函数标记：使用__attribute__((hot))标记热点函数，优化编译
- 分支预测优化：if (likely(ptr != nullptr)) 让CPU更好地预测分支
- 避免伪共享：Atomic计数器之间添加padding

**实现要点**：
1. 线程绑定CPU核心，减少上下文切换
2. 优化热点代码（热点路径内联）
3. 使用编译器优化选项（-O3）
4. 避免伪共享（padding对齐）

**优化方向**：
```
1. SubReactor线程绑定固定CPU核心
2. 热点函数使用__attribute__((hot))标记
3. 使用likely/unlikely分支预测优化
```

**验收标准**：
- 上下文切换减少
- 单请求处理延迟降低
