# 第二章：应用层（命令系统）

## 章节预告

上一章我们看到，当用户输入 `SET name zhangsan` 时，协议层解析出了命令参数数组 `["SET", "name", "zhangsan"]`。

现在问题是：**服务器如何知道"SET"命令是什么意思？如何执行它？**

这就是**应用层**要解决的问题。

---

## 2.1 问题：如何设计命令系统？

### 2.1.1 最简单但最糟糕的方案

```cpp
// 服务器主循环中
if (cmd == "GET") {
    // GET 逻辑
    auto result = storage.get(key);
    send(response);
}
else if (cmd == "SET") {
    // SET 逻辑
    storage.set(key, value);
    send(response);
}
else if (cmd == "DEL") {
    // DEL 逻辑
    storage.del(key);
    send(response);
}
// ... 又要加新命令？继续写 if-else ...
else if (cmd == "EXISTS") {
    // EXISTS 逻辑
    // ...
}
```

**问题**：
```
1. 代码膨胀：100个命令 = 100个 if-else
2. 难以维护：改一个命令要找到对应的 if-else 块
3. 难以测试：所有逻辑混在一起
4. 违反开闭原则：添加新命令要修改已有代码
```

### 2.1.2 设计模式：命令模式

**命令模式**是解决这个问题的完美方案：

```
核心思想：
把"命令"封装成"对象"

命令 = 谁（接收者）+ 做什么（执行动作）
```

```cpp
// 伪代码
class Command {
    virtual string execute(args) = 0;  // 执行命令
};

// GET 命令
class GetCommand : public Command {
    string execute(args) override {
        return storage.get(args[1]);
    }
};

// SET 命令
class SetCommand : public Command {
    string execute(args) override {
        storage.set(args[1], args[2]);
        return "OK";
    }
};
```

**好处**：
```
1. 每个命令独立一个类，代码简洁
2. 添加新命令只需加新类，不动已有代码
3. 可以存储命令、撤销命令、日志命令
4. 易于单元测试
```

## 2.2 命令模式实现

### 2.2.1 Command 基类

**文件：** `src/command/command.h`

```cpp
#ifndef CONCURRENTCACHE_COMMAND_COMMAND_H
#define CONCURRENTCACHE_COMMAND_COMMAND_H

#include <string>
#include <vector>

namespace cc_server {

    /**
     * @brief Command 基类 - 所有命令的抽象接口
     *
     * 设计模式：命令模式（Command Pattern）
     *
     * 什么是命令模式？
     * - 把"请求"封装成"对象"
     * - 请求 = 一个 Redis 命令（如 GET, SET, DEL）
     * - 对象 = Command 子类的实例
     *
     * 为什么要用命令模式？
     * 1. 命令的执行逻辑封装在命令对象里
     * 2. 可以存储命令、撤销命令、队列化命令
     * 3. 易于扩展新命令（添加新子类即可）
     *
     * @param args 命令参数，格式为 [命令名, 参数1, 参数2, ...]
     *             例如：["GET", "key"] 或 ["SET", "key", "value"]
     * @return RESP 格式的响应字符串
     */
    class Command {
    public:
        virtual ~Command() = default;

        /**
         * @brief 执行命令
         * @param args 命令参数向量
         * @return RESP 格式的响应字符串
         *
         * 示例：
         *   SetCommand cmd;
         *   std::string resp = cmd.execute({"SET", "name", "zhangsan"});
         *   // resp = "+OK\r\n"
         */
        virtual std::string execute(const std::vector<std::string>& args) = 0;

        /** @brief 克隆命令对象，用于工厂模式创建实例 */
        virtual std::unique_ptr<Command> clone() const = 0;
    };
}

#endif
```

**设计解析**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            Command 基类设计                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  为什么是抽象基类（有纯虚函数）？                                            │
│  - 定义命令的统一接口                                                       │
│  - 告诉所有子类："你必须实现 execute() 方法"                                │
│                                                                             │
│  为什么 execute() 返回 string 而不是直接发送？                               │
│  - 命令只负责"执行"，不负责"发送"                                         │
│  - 发送是网络层的职责                                                      │
│  - 解耦：命令逻辑和网络逻辑分开                                             │
│                                                                             │
│  为什么要 clone() 方法？                                                    │
│  - 工厂创建的是命令的"模板"                                                │
│  - 每次请求需要创建新的命令实例                                             │
│  - clone() 负责复制自身                                                    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2.2 命令实现：GET 命令

**文件：** `src/command/string_cmd.h`

```cpp
/**
 * @brief GetCommand - GET 命令实现
 *
 * GET 命令：获取指定 key 的值
 * 语法：GET key
 * 返回：
 *   - 如果 key 存在：返回其值（Bulk String 格式）
 *   - 如果 key 不存在：返回 nil
 *
 * RESP 格式：
 *   GET key      → $5\r\nhello\r\n  或  $-1\r\n（不存在）
 */
class GetCommand : public Command {
public:
    std::string execute(const std::vector<std::string>& args) override {
        // 步骤1：参数验证
        // GET 命令需要 2 个参数 [命令名, key]
        // args[0] = "GET", args[1] = key
        if (args.size() != 2) {
            return RespEncoder::encode_error(
                "ERR wrong number of arguments for 'get' command"
            );
        }

        // 步骤2：从存储获取值
        const std::string& key = args[1];
        auto result = GlobalStorage::instance().get(key);

        // 步骤3：根据结果返回不同响应
        // key 不存在时返回 nil
        if (!result.has_value()) {
            return RespEncoder::encode_nil();  // "$-1\r\n"
        }

        // key 存在，返回值
        return RespEncoder::encode_bulk_string(result.value());  // "$5\r\nhello\r\n"
    }

    std::unique_ptr<Command> clone() const override {
        return std::make_unique<GetCommand>(*this);
    }
};
```

### 2.2.3 命令实现：SET 命令

```cpp
/**
 * @brief SetCommand - SET 命令实现
 *
 * SET 命令：设置指定 key 的值
 * 语法：SET key value
 * 返回：
 *   - 成功：OK
 *
 * RESP 格式：
 *   SET key value → +OK\r\n
 */
class SetCommand : public Command {
public:
    std::string execute(const std::vector<std::string>& args) override {
        // 步骤1：参数验证
        // SET 命令需要 3 个参数 [命令名, key, value]
        if (args.size() != 3) {
            return RespEncoder::encode_error(
                "ERR wrong number of arguments for 'set' command"
            );
        }

        // 步骤2：调用存储
        const std::string& key = args[1];
        const std::string& value = args[2];
        GlobalStorage::instance().set(key, value);

        // 步骤3：返回成功响应
        return RespEncoder::encode_ok();  // "+OK\r\n"
    }

    std::unique_ptr<Command> clone() const override {
        return std::make_unique<SetCommand>(*this);
    }
};
```

/**
 * @brief DelCommand - DEL 命令实现
 *
 * DEL 命令：删除指定 key
 * 语法：DEL key

 *   - key 不存在：0
 *

    }
};

## 2.3 命令工厂

### 2.3.1 为什么需要工厂？

**不使用工厂**：

```cpp
// 每次请求都要 new
if (cmd == "get") {
    auto cmd = new GetCommand();  // 内存泄漏风险！
    response = cmd->execute(args);
    delete cmd;
}
```

**使用工厂**：

```cpp
// 工厂创建，自动管理内存
auto cmd = CommandFactory::instance().create("get");
if (cmd) {
    response = cmd->execute(args);
    // cmd 是 unique_ptr，出了作用域自动释放
}
```

### 2.3.2 工厂实现

**文件：** `src/command/command_factory.h`

```cpp
#ifndef CONCURRENTCACHE_COMMAND_COMMAND_FACTORY_H
#define CONCURRENTCACHE_COMMAND_COMMAND_FACTORY_H

#include <memory>
#include <unordered_map>
#include "command.h"

namespace cc_server {

    /**
     * @brief CommandFactory - 命令工厂类
     *
     * 工厂模式：根据命令名创建对应的命令对象
     *
     * 为什么需要工厂？
     * - 简单做法：if (cmd == "get") return new GetCommand();
     * - 问题：如果有很多命令，if-else 会很长
     * - 工厂模式：用 map 存储映射，添加新命令只需修改注册部分
     *
     * 使用方法：
     *   CommandFactory::instance().register_command("get", std::make_unique<GetCommand>());
     *   auto cmd = CommandFactory::instance().create("get");
     */
    class CommandFactory {
    public:
        /**
         * @brief 获取单例实例
         *
         * 使用 Magic Static 保证线程安全
         */
        static CommandFactory& instance();

        /**
         * @brief 注册命令
         * @param name 命令名（如 "get", "set"）
         * @param cmd 命令对象（智能指针）
         *
         * 示例：
         *   factory.register_command("get", std::make_unique<GetCommand>());
         */
        void register_command(const std::string& name, std::unique_ptr<Command> cmd);

        /**
         * @brief 创建命令对象
         * @param name 命令名
         * @return 命令对象指针，未找到返回 nullptr
         *
         * 示例：
         *   auto cmd = factory.create("get");
         *   if (cmd) {
         *       std::string resp = cmd->execute({"GET", "key"});
         *   }
         */
        std::unique_ptr<Command> create(const std::string& name);

        // 禁用拷贝，确保单例唯一性
        CommandFactory(const CommandFactory&) = delete;
        CommandFactory& operator=(const CommandFactory&) = delete;

    private:
        /**
         * @brief 私有构造函数
         *
         * 在构造函数中注册所有内置命令
         */
        CommandFactory();

        // 命令注册表：命令名 → 命令对象
        std::unordered_map<std::string, std::unique_ptr<Command>> commands_;
    };
}
#endif
```

**文件：** `src/command/command_factory.cpp`

```cpp
#include "command_factory.h"
#include "string_cmd.h"

namespace cc_server {

    // =========================================================================
    // 私有构造函数：自动注册所有内置命令
    // =========================================================================
    //
    // 为什么在这里注册命令？
    // - 每次创建 CommandFactory 实例时，自动注册所有支持的命令
    // - 由于是单例模式，只会有一个实例，所以只注册一次
    //
    CommandFactory::CommandFactory() {
        // 注册内置命令
        // std::make_unique<GetCommand>() 等同于 new GetCommand() 但更安全
        register_command("get", std::make_unique<GetCommand>());
        register_command("set", std::make_unique<SetCommand>());
        register_command("del", std::make_unique<DelCommand>());
        register_command("exists", std::make_unique<ExistsCommand>());

        // 注意：Redis 命令不区分大小写，但通常用小写存储
        // 可以在协议层做大小写转换
    }

    // =========================================================================
    // instance() - 获取单例实例
    // =========================================================================
    //
    // 实现：Magic Static（C++11 线程安全局部静态变量）
    //
    CommandFactory& CommandFactory::instance() {
        static CommandFactory instance;  // 只初始化一次，线程安全
        return instance;
    }

    // =========================================================================
    // register_command() - 注册命令
    // =========================================================================
    //
    void CommandFactory::register_command(const std::string& name,
                                          std::unique_ptr<Command> cmd) {
        // std::move 转移 unique_ptr 的所有权
        commands_[name] = std::move(cmd);
    }

    // =========================================================================
    // create() - 创建命令对象
    // =========================================================================
    //
    std::unique_ptr<Command> CommandFactory::create(const std::string& name) {
        // 在 map 中查找命令
        auto it = commands_.find(name);

        if (it != commands_.end()) {
            // 找到命令，使用 clone() 创建新实例
            return it->second->clone();
        }

        // 命令不存在
        return nullptr;
    }
}
```

## 2.4 命令执行流程

### 2.4.1 main.cpp 中的命令处理

**文件：** `main.cpp`

```cpp
int main() {
    // ... 初始化 ...

    // 创建 Connection 后，设置命令处理回调
    auto* conn = new Connection(client_fd, &loop);
    conn->set_command_callback([](const RespValue& cmd, Connection* conn) {
        // ===== 命令处理核心逻辑 =====

        // 1. 从 RespValue 提取命令数组
        const auto& arr = cmd.as_array();
        if (arr.empty()) {
            return;  // 空命令，忽略
        }

        // 2. 提取命令参数
        std::vector<std::string> args;
        for (size_t i = 0; i < arr.size(); ++i) {
            args.push_back(arr[i].as_string());
        }

        // 3. 命令名转小写（Redis 协议不区分大小写）
        if (!args.empty()) {
            std::string& cmd_name = args[0];
            std::transform(cmd_name.begin(), cmd_name.end(), cmd_name.begin(),
                           [](unsigned char c) { return std::tolower(c); });
        }

        // 4. 创建并执行命令
        auto command = CommandFactory::instance().create(args[0]);
        if (command) {
            std::string response = command->execute(args);
            conn->send_response(response);  // 发送响应
        } else {
            // 命令不存在
            conn->send_response(RespEncoder::encode_error("ERR unknown command"));
        }
    });

    // ...
}
```

## 2.5 为什么这样设计？

### 2.5.1 设计原则

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           设计原则与权衡                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  1. 单一职责原则（SRP）                                                     │
│     ─────────────────────                                                    │
│     每个类只做一件事：                                                       │
│     - Command: 只负责命令执行                                               │
│     - CommandFactory: 只负责创建命令                                        │
│     - Storage: 只负责数据存储                                               │
│                                                                             │
│  2. 开闭原则（OCP）                                                         │
│     ─────────────────                                                      │
│     对扩展开放，对修改关闭：                                                 │
│     - 添加新命令：只需实现新 Command 类 + 注册                               │
│     - 不需要修改已有命令代码                                                 │
│                                                                             │
│  3. 依赖倒置原则（DIP）                                                     │
│     ─────────────────────                                                  │
│     依赖抽象，不依赖具体：                                                   │
│     - main.cpp 依赖 Command 抽象，不依赖具体命令                            │
│     - 具体命令可以随时替换                                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.5.2 权衡取舍

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           设计权衡                                           │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  选择命令模式的原因：                                                        │
│  ✓ 代码组织清晰，每命令一类                                                │
│  ✓ 易于扩展新命令                                                         │
│  ✓ 易于测试（每个命令独立）                                                │
│  ✗ 多了一点间接层（轻微性能开销，可忽略）                                   │
│                                                                             │
│  选择工厂模式的原因：                                                        │
│  ✓ 统一管理命令创建                                                        │
│  ✓ 可以做命令验证、装饰等                                                  │
│  ✓ 配合单例模式，全局唯一                                                  │
│  ✗ 需要理解工厂概念                                                        │
│                                                                             │
│  选择不做的权衡：                                                           │
│  - 没有实现命令参数校验框架 → 简单够用                                      │
│  - 没有实现命令装饰器 → 后续版本再加                                       │
│  - 没有实现命令撤销 → 缓存服务器不需要                                     │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 2.6 本章小结

### 6.1 关键概念

| 概念 | 说明 |
|------|------|
| 命令模式 | 把请求封装成对象，便于扩展和管理 |
| 工厂模式 | 统一创建对象，隐藏创建细节 |
| 单例模式 | 全局只有一个实例 |

### 6.2 已实现命令

| 命令 | 语法 | RESP 返回 |
|------|------|-----------|
| GET | `GET key` | Bulk String 或 Nil |
| SET | `SET key value` | OK |
| DEL | `DEL key` | Integer (删除数量) |
| EXISTS | `EXISTS key` | Integer (存在数量) |

### 6.3 扩展命令的步骤

```
1. 在 string_cmd.h 中实现新命令类
2. 在 command_factory.cpp 中注册命令
3. 重新编译
```
