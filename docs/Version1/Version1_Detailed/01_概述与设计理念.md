# 第一章：概述与设计理念

## 1.1 项目背景

### 1.1.1 为什么要开发这个项目？

在互联网后端开发中，**缓存**是提升系统性能的关键技术。Redis 作为最流行的内存数据库，被广泛应用于各大互联网公司。

```
没有缓存：
用户请求 ──► 数据库查询 ──► 返回结果
                    │
                    ▼ 慢！数据库压力大！

有缓存：
用户请求 ──► 缓存查找 ──► 命中！返回
                              │
                              ▼ 未命中
                        数据库查询 ──► 写入缓存 ──► 返回
```

但是有时候：
1. **只是想学习 Redis 的工作原理**
2. **项目太小，不需要引入完整的 Redis**
3. **想要一个可以嵌入的轻量级缓存**

于是你决定：**自己写一个简单的缓存服务器**。

### 1.1.3 约束与挑战

作为骨架版本（V1），我们面临一些约束：

| 约束 | 说明 | 应对 |
|------|------|------|
| 单线程 Reactor | 只能使用一个 CPU 核心 | 先实现功能，后续版本升级多线程 |
| 简单存储 | 只用 unordered_map | 先能用就行，后续版本加 TTL/LRU |

**设计原则**：先解决"能不能跑"的问题，再解决"跑得好不好"的问题。

## 1.3 核心设计理念

### 1.3.1 为什么选择事件驱动而不是多线程？

**服务器处理请求的方式对比：**

```
方式一：每请求一线程（Thread-per-connection）

    主线程                    线程池
       │                        │
       ▼                        ▼
    接受连接 ──────────────► 分配线程
       │                        │
       │                        ▼
       │                   处理请求
       │                        │
       │                        ▼
       │                   返回响应
       │
       ▼
    接受下一个连接...

    问题：
    - 1000个连接 = 1000个线程
    - 线程占用栈内存（通常2MB）
    - 1000线程 × 2MB = 2GB 内存！
    - 线程切换有开销
```

```
方式二：事件驱动（Event-driven）

    EventLoop (单线程)
           │
           ▼
    epoll_wait() ── 阻塞等待
           │
           ├── [连接A可读] ── 处理A ── epoll_wait()
           ├── [连接B可写] ── 处理B ── epoll_wait()
           └── [连接C可读] ── 处理C ── epoll_wait()

    优点：
    - 单线程处理所有连接
    - 没有线程创建/销毁开销
    - 没有锁竞争
    - 内存占用极低

    缺点：
    - 单核利用
    - CPU密集型任务会阻塞
```

**结论**：

| 场景 | 推荐方案 |
|------|----------|
| IO 密集型（网络服务器） | 事件驱动 ✓ |
| CPU 密集型（计算任务） | 多线程 |
| 混合型 | 事件驱动 + 线程池 |

网络服务器大部分时间在等待 IO（等数据、发送数据），所以**事件驱动是最优选择**。

```
不分层的问题：

┌─────────────────────────────────────────┐
│  main() {                             │
│      listen_fd = socket();             │
│      bind(listen_fd, port);            │
│      listen(listen_fd);                │
```

```
│  - 如何理解 TCP 字节流？                │
│  - 解析 RESP 协议                      │
└───────────────┬─────────────────────────┘
                │ 结构化数据
                ▼
┌─────────────────────────────────────────┐
│  存储层：数据管理                        │
│  - 数据存在哪？                         │
│  - 如何保证线程安全？                   │
└───────────────┬─────────────────────────┘
                │ 存储操作
                ▼
┌─────────────────────────────────────────┐
│  网络层：通信传输                        │
```

### 1.3.3 为什么要用 RAII 和智能指针？

**C++ 资源管理的常见问题**：

```cpp
// 问题代码：内存泄漏
void handle() {
    Connection* conn = new Connection(fd);  // 分配内存
    // ... 处理逻辑 ...
    if (error) {
        return;  // 糟糕！忘记 delete，内存泄漏！
    }
    // ... 更多代码 ...
    delete conn;  // 可能执行不到
}

// 问题代码：重复释放
void handle() {
    Connection* conn = new Connection(fd);
    // ... 处理逻辑 ...
    delete conn;  // 第一次删除
    // ... 更多代码 ...
    delete conn;  // 重复删除！崩溃！
}
```

**RAII 解决方案**：

```cpp
// 安全代码：RAII + 智能指针
void handle() {
    auto conn = std::make_unique<Connection>(fd);  // 自动管理
    // ... 处理逻辑 ...
    if (error) {
        return;  // conn 自动销毁，内存安全释放
    }
    // ... 更多代码 ...
    return;  // conn 自动销毁
}
```

**RAII 原理**：
```
创建对象时 ──► 构造函数申请资源
销毁对象时 ──► 析构函数释放资源
对象的生命周期 ──► 绑定到作用域
```

## 1.6 版本1 能力与局限

### 1.6.1 已实现功能

| 功能 | 状态 | 说明 |
|------|------|------|
| TCP 监听 | ✅ | 支持端口监听、接受连接 |
| 非阻塞 IO | ✅ | SOCK_NONBLOCK 模式 |
| epoll 事件驱动 | ✅ | 单 Reactor 模型 |
| RESP 协议解析 | ✅ | 支持 5 种数据类型 |
| 命令系统 | ✅ | GET/SET/DEL/EXISTS |
| 线程安全存储 | ✅ | shared_mutex 读写锁 |
| 基础日志 | ✅ | INFO/DEBUG/ERROR 级别 |

### 1.6.2 存在的不足

| 不足 | 影响 | 后续版本解决 |
|------|------|-------------|
| 单线程 Reactor | 无法利用多核 CPU | V2: MainSubReactor |
| 全局一把锁 | 高并发时锁竞争严重 | V2: 分段锁哈希表 |
| 无内存池 | malloc/free 开销大 | V2: 三层内存池 |
| 无连接池 | 频繁创建销毁连接 | V2: 连接池 |
| 无 TTL | 数据永不过期 | V2: ExpireDict |
| 无 LRU | 内存无限增长 | V2: LRU 算法 |
| 命令有限 | 只有 4 个命令 | V2: 完整字符串命令 |

### 1.6.3 性能数据

```
测试环境：单核 2.5GHz CPU，8GB 内存

简单 GET/SET 测试：
- 并发 100 连接：约 50,000 QPS
- 并发 500 连接：约 60,000 QPS（单核瓶颈）

瓶颈原因：
1. 单线程只能用一个核心
2. 全局锁竞争
3. 没有内存池优化
```
