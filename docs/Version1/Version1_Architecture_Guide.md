## 故事线：为什么要开发这个项目？

### 起点：遇到的问题

假设你是一个后端工程师，你的系统需要缓存热点数据来提升性能。你听说 Redis 很好用，于是你想：

```
系统 A (Go) ──────► Redis (6379) ──────► 存储
系统 B (Python) ───► Redis (6379) ──────► 存储
系统 C (Java) ────► Redis (6379) ──────► 存储
```

但是问题来了：
1. **Redis 太重了**：你只是想要一个简单的键值缓存，不需要 Redis 的全部功能
2. **运维成本**：需要单独部署和维护 Redis 进程
3. **学习目的**：你想深入理解高并发服务器是怎么工作的

于是你决定：**自己写一个简单的缓存服务器**。

### 版本1 的目标

```
目标：开发一个最小可运行的 Redis 风格缓存服务器

功能需求：
□ 接受 TCP 连接
□ 解析类似 Redis 的命令协议
□ 支持基本的键值操作（GET/SET/DEL/EXISTS）
□ 线程安全地存储数据
□ 高性能、高并发

质量需求：
□ 代码结构清晰，易于理解
□ 模块化设计，便于扩展
□ 注释详尽，作为学习资料
```

## 核心设计理念

### 1. 事件驱动 vs 多线程

**为什么选择事件驱动（EventLoop + epoll）而不是多线程？**

```
多线程方案：
┌─────────┐ ┌─────────┐ ┌─────────┐
│ 线程1   │ │ 线程2   │ │ 线程3   │   ← 每个线程都要占用栈内存
│ 处理A   │ │ 处理B   │ │ 处理C   │
│ 等待I/O │ │ 等待I/O │ │ 等待I/O │   ← 大部分时间在等待，浪费CPU
└─────────┘ └─────────┘ └─────────┘

事件驱动方案：
┌─────────────────────────────────────┐
│           EventLoop (单线程)        │
│                                     │
│   epoll_wait() 阻塞等待 ──────► 有事件才处理
│                                     │
│   处理 A ──► 处理 B ──► 处理 C       │  ← CPU 始终在工作
└─────────────────────────────────────┘
```

**结论**：对于 IO 密集型任务（网络服务器大部分时间在等数据），事件驱动更高效。

### 2. 为什么需要模块化分层？

```
不分层的问题：
┌─────────────────────────────────────────┐
│  main() {                             │
│      accept();                         │
│      parse_command();                  │
│      execute_storage();                 │
│      send_response();                  │
│  }                                    │
│                                         │
│  问题：所有逻辑混在一起，改一个功能要 │
│  理解全部代码，难以维护和扩展          │
└─────────────────────────────────────────┘

分层的优势：
┌─────────────────────────────────────────┐
│  应用层：命令处理（用户能看到的）       │
├─────────────────────────────────────────┤
│  协议层：数据解析（如何理解数据）       │
├─────────────────────────────────────────┤
│  存储层：数据存储（如何保存数据）       │
├─────────────────────────────────────────┤
│  网络层：通信传输（如何收发数据）       │
└─────────────────────────────────────────┘

优势：
- 每层只关注自己的事
- 可以独立测试、替换
- 易于理解和维护
```

### 3. 为什么要用 RAII 和智能指针？

```
手动管理内存的问题：
void handle_request() {
    Connection* conn = new Connection(fd);
    // ... 处理逻辑 ...
    if (error) {
        delete conn;  // 忘记 delete？内存泄漏！
        return;
    }
    // ... 更多代码 ...
    delete conn;  // 重复 delete？崩溃！
}

智能指针方案：
void handle_request() {
    auto conn = std::make_unique<Connection>(fd);
    // ... 处理逻辑 ...
    if (error) {
        return;  // conn 自动销毁
    }
    // ... 更多代码 ...
    return;  // conn 自动销毁
}
```
